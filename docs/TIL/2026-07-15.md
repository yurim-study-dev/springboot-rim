### 1. Thymeleaf 레이아웃 - Fragment

타임리프의 Fragment 기능으로 공통 UI를 분리해서 재사용할 수 있다.

#### Fragment 정의 (`layout/common.html`)
```html
<th:block th:fragment="nav-header(title)">
    <h1 th:text="${title}">타이틀</h1>
    <div class="nav">
        <a th:href="@{write.html}">새글</a>
        <a th:href="@{list.html}">목록으로 이동</a>
    </div>
</th:block>
```
- `th:fragment="이름(파라미터)"` 로 조각을 선언
- `th:block` 은 렌더링 시 HTML 태그 없이 사라지는 가상 태그

#### Fragment 사용 (`list.html`, `write.html`)
```html
<!-- th:insert: 해당 태그 안에 fragment를 삽입 -->
<th:block th:insert="~{layout/common :: nav-header('게시글 목록')}">
    ...
</th:block>

<!-- 동적 파라미터 전달 -->
<th:block th:insert="~{layout/common :: nav-header(${postForm.id == 0} ? '게시글 등록' : '게시글 수정')}">
```

**`th:insert` vs `th:replace` 차이**
| 속성 | 결과 |
|------|------|
| `th:insert` | 현재 태그는 유지되고, 그 안에 fragment가 삽입됨 |
| `th:replace` | 현재 태그 자체가 fragment로 대체됨 |

**Fragment 참조 문법**: `~{파일경로 :: fragment이름(파라미터)}`

### 2. Layered Architecture (계층형 아키텍처)

board03에서 Controller에 몰려있던 로직을 각 계층으로 분리했다.

```
요청 → Controller → Service → Repository → (데이터)
                 ← Service ← Repository ←
응답 ← Controller
```

**각 계층의 역할**
- **Controller**: HTTP 요청/응답 처리, View와 소통
- **Service**: 비즈니스 로직 담당
- **Repository**: 데이터 접근 담당 (DB, 메모리 등)

#### Interface + Impl 패턴

각 계층을 **인터페이스**와 **구현체**로 분리한다.

```java
// Service 인터페이스
public interface PostService {
    List<PostDto> getPosts();
    PostDto getPost(int id);
    void writePost(PostDto post);
    void editPost(PostDto post);
    void removePost(int id);
}

// Service 구현체
@Service
public class PostServiceImpl implements PostService {
    private final PostRepository postRepository;

    public PostServiceImpl(PostRepository postRepository) {
        this.postRepository = postRepository;
    }

    @Override
    public List<PostDto> getPosts() { return postRepository.findAll(); }

    @Override
    public void writePost(PostDto post) { postRepository.save(post); }
    // ...
}
```

```java
// Repository 인터페이스
public interface PostRepository {
    List<PostDto> findAll();
    PostDto findById(int id);
    void save(PostDto post);
    void update(PostDto post);
    void deleteById(int id);
}

// Repository 구현체 (메모리 저장)
@Repository
public class MemoryPostRepository implements PostRepository {
    private final List<PostDto> fakePosts;

    @Override
    public PostDto findById(int id) {
        for (PostDto org : fakePosts) {
            if (org.getId() == id) return org;
        }
        throw new IllegalArgumentException(id + "번 게시글은 존재하지 않습니다.");
    }
    // ...
}
```

**인터페이스를 쓰는 이유**: 나중에 메모리 저장 → DB 저장으로 바꿀 때 Controller/Service 코드를 건드리지 않고 구현체(MemoryPostRepository → JpaPostRepository)만 교체하면 된다.

### 3. @Valid + BindingResult (유효성 검사)

#### DTO에 검증 어노테이션 추가
```java
public class PostDto {
    @NotBlank(message = "제목은 필수 입력 항목입니다.")
    @Size(max = 100, message = "제목은 100자 이하로 입력해야 합니다.")
    private String title;

    @NotBlank(message = "내용은 필수 입력 항목입니다.")
    private String content;

    @NotBlank(message = "작성자는 필수 입력 항목입니다.")
    @Size(min = 2, max = 10, message = "작성자 이름은 2자 이상 10자 이하여야 합니다.")
    private String author;
}
```

#### Controller에서 검증 처리
```java
@PostMapping("/write")
public String writePost(
        @Valid @ModelAttribute("postForm") PostDto post,
        BindingResult bindingResult) {

    if (bindingResult.hasErrors()) {
        return "board/write"; // 에러 있으면 폼 페이지로 다시
    }
    postService.writePost(post);
    return "redirect:list.html";
}
```
- `@Valid`: 파라미터 검증 실행
- `BindingResult`: 검증 결과 저장 객체, **반드시 `@Valid` 파라미터 바로 다음에 위치**
- `bindingResult.hasErrors()` 가 true면 저장하지 않고 폼으로 돌아감

#### 타임리프에서 에러 메시지 출력
```html
<input type="text" th:field="*{title}">
<div class="field-error" 
     th:if="${#fields.hasErrors('title')}" 
     th:errors="*{title}">에러메시지</div>
```
- `#fields.hasErrors('필드명')`: 해당 필드에 에러가 있는지 확인
- `th:errors="*{필드명}"`: 에러 메시지를 자동으로 출력

### 4. GlobalExceptionHandler (@ControllerAdvice)

예외 처리를 각 Controller마다 따로 하는 게 아니라, 한 곳에서 전역으로 처리한다.

```java
@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(IllegalArgumentException.class)
    public String handleIllegalArgumentException(
            IllegalArgumentException ex, Model model) {
        model.addAttribute("errorMessage", ex.getMessage());
        return "error/400"; // templates/error/400.html 렌더링
    }
}
```
- `@ControllerAdvice`: 모든 Controller에 적용되는 전역 처리 클래스
- `@ExceptionHandler(예외.class)`: 해당 예외 발생 시 이 메서드로 처리
- Repository에서 던진 `IllegalArgumentException`이 어디서 발생하든 자동으로 잡힘

예: 없는 게시글 ID 조회 → `MemoryPostRepository.findById()`에서 예외 발생 → `GlobalExceptionHandler`가 잡아서 에러 페이지 렌더링

### 5. DispatcherServlet 요청 처리 흐름

```
① 클라이언트 요청 (HTTP Request)
         ↓
② DispatcherServlet (모든 요청의 입구)
         ↓
③ HandlerMapping (어떤 Controller의 어떤 메서드가 처리할지 결정)
         ↓
④ HandlerAdapter (Controller 메서드 실행)
         ↓
⑤ Controller (비즈니스 로직 실행, ModelAndView 반환)
         ↓
⑥ ViewResolver (논리 뷰 이름 → 실제 파일 경로로 변환)
         ↓
⑦ View (HTML 렌더링)
         ↓
⑧ 클라이언트 응답 (HTTP Response)
```

- **DispatcherServlet**: Spring MVC의 핵심 서블릿, 모든 요청을 중앙에서 처리 (Front Controller 패턴)
- **ViewResolver**: `"board/write"` → `templates/board/write.html` 로 매핑

### 6. @ModelAttribute

```java
@GetMapping("/write.html")
public String getWriteForm(@ModelAttribute("postForm") PostDto post) {
    return "board/write";
}
```
- `@ModelAttribute`: 빈 DTO 객체를 자동으로 생성해서 Model에 등록
- `"postForm"` 이름으로 View에서 `${postForm}` 으로 접근 가능
- 폼 태그에서 `th:object="${postForm}"` 으로 바인딩할 때 필요
