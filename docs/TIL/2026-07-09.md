## 1. DI 주입 방식 비교 (`spring/di`)

스프링이 의존성을 주입하는 방법이 여러 가지인데, 오늘은 두 가지 실습했다.

### 생성자 주입 (`di/constructor`)

```java
// Driver.java
public class Driver {
    private final Car car;

    Driver(Car car) {  // 생성자로 받음
        this.car = car;
    }
}

// AppConfig.java
@Bean
public Driver driver(Car car) {
    return new Driver(car);  // 생성자에 넣어서 생성
}
```

- 객체 만들 때 딱 한 번만 주입되고 이후 변경 불가 → **안전**
- `final` 키워드 쓸 수 있어서 주입 안 하면 컴파일 오류로 바로 알 수 있음
- **실무 표준 방식**

### 세터 주입 (`di/setter`)

```java
// Driver.java
public class Driver {
    private Car car;

    public void setCar(Car car) {  // setter로 받음
        this.car = car;
    }
}

// AppConfig.java
@Bean
public Driver driver(Car car) {
    Driver driver = new Driver();
    driver.setCar(car);  // setter로 주입
    return driver;
}
```

- 나중에 car를 바꿀 수 있음 → 유연하지만 주입 빠뜨리면 NullPointerException 위험
- 주의: `driver.setCar(car)` 빼먹으면 car가 null이 돼서 `driveCar()` 실행 시 터짐

## 2. AOP - 왜 필요한가

입금/출금/이체 기능 예시:

```
로그            ← 3개 다 똑같이 필요 (횡단 관심사)
인증            ← 3개 다 똑같이 필요
트랜잭션 시작    ← 3개 다 똑같이 필요
─────────────────────────────
입금 로직 / 출금 로직 / 이체 로직  ← 핵심 관심사 (얘네만 다름)
─────────────────────────────
트랜잭션 종료    ← 3개 다 똑같이 필요
로그            ← 3개 다 똑같이 필요
예외처리         ← 3개 다 똑같이 필요
```

반복 코드를 메서드마다 복붙하지 않고 한 곳에서 관리하는 게 AOP

## 3. AOP 구현 방법 3단계 (`spring/aop`)

### 1단계: 정적 프록시 (`staticproxy`)

Car를 감싸는 LogProxy 클래스를 직접 만들어서 앞뒤에 코드 끼워넣기

```java
public class LogProxy implements Car {
    private final Car target;  // 진짜 Car

    @Override
    public void drive() {
        System.out.println("[실행 전] 안전벨트를 맵니다.");
        target.drive();  // 진짜 drive 실행
        System.out.println("[실행 후] 안전벨트를 풉니다.");
    }
}
```

- 문제점: Car 메서드마다 LogProxy에 일일이 구현해야 함. 메서드 100개면 100번 작성

### 2단계: 동적 프록시 (`dynamicproxy`)

자바가 런타임에 프록시를 자동 생성해줌. 코드 하나로 모든 메서드에 적용

```java
public class TimeCheckInvocationHandler implements InvocationHandler {
    private final Car target;

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("[동적 프록시] 실행 전: " + method.getName());
        long start = System.currentTimeMillis();
        Object result = method.invoke(target, args);  // 진짜 메서드 실행
        long end = System.currentTimeMillis();
        System.out.println("[동적 프록시] 실행 후: " + (end - start) + "ms");
        return result;
    }
}
```

- 개선: 메서드 하나로 모든 메서드 커버
- 문제점: 인터페이스(Car)가 있어야만 사용 가능

### 3단계: 스프링 AOP (`springaop`)

스프링이 프록시 생성을 전부 자동으로 해줌. 어노테이션만 붙이면 끝

```java
@Aspect
public class LoggingAspect {

    @Before("execution(* *Driver.*(..))")
    public void logBefore() {
        System.out.println("[AOP] 메서드 실행 전");
    }

    @After("execution(* *Driver.*(..))")
    public void logAfter() {
        System.out.println("[AOP] 메서드 실행 후");
    }

    @Around("execution(* *Driver.*(..))")
    public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
        System.out.println("[AOP Around] 전");
        Object result = joinPoint.proceed();  // 실제 메서드 실행
        System.out.println("[AOP Around] 후");
        return result;
    }
}
```

## 4. AOP 핵심 용어 정리

| 용어 | 한 줄 요약 |
|------|----------|
| Aspect | 횡단 관심사 모아둔 클래스 (`LoggingAspect`) |
| Target | Aspect 적용 대상 (`Driver`) |
| Pointcut | 어디에 적용할지 필터 (`execution(* *Driver.*(..))`) |
| Advice | 실제 실행할 코드 (`@Before`, `@After`, `@Around`) |
| JoinPoint | Advice가 끼어들 수 있는 지점 (메서드 실행 시점) |

### Advice 5종류

| 어노테이션 | 실행 시점 |
|-----------|---------|
| `@Before` | 메서드 실행 전 |
| `@After` | 메서드 실행 후 (성공/실패 상관없이) |
| `@AfterReturning` | 정상 완료 후 |
| `@AfterThrowing` | 예외 발생 시 |
| `@Around` | 전/후 전부 (가장 강력, proceed()로 원본 실행) |

## 느낀 점

정적 프록시 → 동적 프록시 → 스프링 AOP 순서로 보니까 왜 AOP가 생겼는지 이해됐다.
결국 "반복 코드 여기저기 복붙하지 말고 한 곳에서 관리하자"는 거였음.
`@Transactional` 같은 게 이 AOP로 만들어진 거라는 게 신기함.
