# 2026-07-08 TIL - 스프링 프레임워크 핵심 원리 첫 수업

## 오늘 한 것

스프링부트 수업 첫날. 순수 자바 Gradle 프로젝트(SpringCore)에 스프링 의존성을 직접 추가하고, IoC/DI 개념을 `oop` → `spring` 패키지 실습으로 단계별로 따라가봤다.

---

## 1. 프레임워크 vs 라이브러리

| 구분 | 제어 주체 | 예시 |
|------|----------|------|
| 라이브러리 | 개발자가 직접 호출 | 날짜 계산 라이브러리 |
| 프레임워크 | 프레임워크가 내 코드를 호출 | Spring |

핵심 차이는 **제어의 주체**. 프레임워크는 내가 코드를 삽입하면 프레임워크가 알아서 타이밍에 맞게 호출해줌 → 이게 IoC(제어의 역전).

---

## 2. 의존성 주입(DI) - 강한 결합 → 느슨한 결합

### Before (강한 결합): `oop/before`

```java
class Driver {
    private HybridCar car = new HybridCar(); // 차종 바꾸려면 이 코드 직접 수정해야 함
}
```

Driver가 특정 구현체(HybridCar)를 직접 `new`로 생성 → 차종 바꾸려면 Driver 코드를 열어서 수정해야 함.

### After (느슨한 결합): `oop/after`

```java
interface Car { ... }

class Driver {
    private Car car; // 인터페이스에만 의존

    public Driver(Car car) { // 외부에서 주입받음
        this.car = car;
    }
}

// 사용 시
Car car = new HybridCar(); // 여기서만 바꾸면 됨
Driver driver = new Driver(car);
```

Driver는 `Car` 인터페이스만 알고, 어떤 구현체가 들어올지 모름. 차종 교체 시 Driver 코드 건드릴 필요 없음.

---

## 3. 스프링 컨테이너로 DI 자동화: `spring` 패키지

### build.gradle에 의존성 추가

```groovy
dependencies {
    implementation("org.springframework:spring-context:7.0.8")
    ...
}
```

### AppConfig - 빈 등록 설정 클래스

```java
@Configuration
public class AppConfig {
    @Bean
    public Car car() {
        return new HybridCar(); // 여기만 바꾸면 전체 적용
    }

    @Bean
    public Driver driver(Car car) {
        return new Driver(car); // 스프링이 car 빈 자동 주입
    }
}
```

### SpringCoreApplication - 컨테이너 구동

```java
ApplicationContext context = new AnnotationConfigApplicationContext(AppConfig.class);
Driver driver = context.getBean(Driver.class); // 컨테이너에서 꺼냄
driver.driveCar();
```

개발자가 `new Driver(new HybridCar())` 이런 식으로 직접 조립하지 않아도, **스프링 컨테이너가 AppConfig를 보고 알아서 객체 생성 + 의존성 연결**해줌.

---

## 4. 핵심 용어 정리

| 용어 | 한 줄 요약 |
|------|----------|
| IoC | 객체 생성/흐름 제어권을 개발자 → 프레임워크로 넘김 |
| DI | 필요한 객체를 직접 만들지 않고 외부에서 받아 씀 |
| 스프링 빈 | 스프링 컨테이너가 관리하는 객체 |
| 스프링 컨테이너 | 빈 생성, 의존성 연결, 생명주기 관리 담당 |
| `@Configuration` | 이 클래스가 스프링 설정 클래스임을 선언 |
| `@Bean` | 이 메서드의 반환 객체를 스프링 빈으로 등록 |

## 5. 오늘 겪은 트러블슈팅

**Gradle 실행 시 한글 콘솔 출력 깨짐**

- 원인: JDK 17+부터 기본 인코딩이 UTF-8로 바뀌었는데 Windows 터미널은 아직 CP949 사용
- 해결: `build.gradle` 맨 끝에 아래 추가

```groovy
tasks.withType(JavaExec).configureEach {
    jvmArgs '-Dfile.encoding=UTF-8'
    jvmArgs '-Dstdout.encoding=UTF-8'
}
```

## 느낀 점

`oop/before` → `oop/after` → `spring` 순서로 단계적으로 코드가 바뀌는 걸 직접 치면서 "왜 스프링이 필요한가"가 자연스럽게 이해됐다. `AppConfig`에서 구현체 딱 한 줄만 바꾸면 전체 앱 동작이 바뀌는 게 신기함.
