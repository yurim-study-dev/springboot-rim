## 컴포넌트 스캔 (Component Scan)

기존에는 `@Configuration` + `@Bean`으로 빈을 하나씩 등록했는데, 빈이 많아질수록 설정이 커지는 문제가 있었음.  
**컴포넌트 스캔**은 `@Component`가 붙은 클래스를 자동으로 찾아서 빈으로 등록해주는 기능.

```java
@Component               // 빈 이름: gasolineCar (클래스명 첫 글자 소문자)
public class GasolineCar implements Car { }

@Component("hCar")       // 빈 이름을 직접 지정할 수도 있음
public class HybridCar implements Car { }

@Component
public class Driver { }

@Configuration
@ComponentScan(basePackages = "net.likelion.bebc25.spring")
public class AppConfig {
    // @Bean 없어도 됨 — 컴포넌트 스캔이 자동 등록
}
```

- 패키지를 생략하면 설정 파일이 있는 패키지와 그 하위 전체를 스캔
- 빈 이름은 클래스명의 첫 글자를 소문자로 변환 (`GasolineCar` → `gasolineCar`)
- `@Component("이름")`으로 빈 이름을 직접 지정할 수도 있음

## 의존관계 자동 주입 (@Autowired)

`@Autowired`를 붙이면 스프링이 컨테이너에서 타입이 맞는 빈을 찾아서 자동으로 주입해줌.

### 주입 방식 3가지

```java
// 1. 생성자 주입 (권장)
@Autowired
public Driver(Car car) {
    this.car = car;
}
// 생성자가 1개면 @Autowired 생략 가능

// 2. 수정자(Setter) 주입
@Autowired
public void setCar(Car car) {
    this.car = car;
}

// 3. 필드 주입 (간편하지만 테스트 어려움)
@Autowired
private Car car;
```

---

## 동일 타입 빈이 여러 개일 때 충돌 해결

`Car` 타입이 `GasolineCar`, `HybridCar` 두 개면 `NoUniqueBeanDefinitionException` 예외 발생.

```java
// 방법 1: @Primary — 기본값 지정
@Component("hCar")
@Primary
public class HybridCar implements Car { }  // 충돌 시 HybridCar가 먼저 주입됨

// 방법 2: @Qualifier — 주입 지점에서 직접 지정
@Autowired
public Driver(@Qualifier("gasolineCar") Car car) {
    this.car = car;
}

// 실습에서 Driver는 필드 주입 방식 사용
@Component
public class Driver {
    @Autowired
    private Car car;  // @Primary인 HybridCar가 주입됨
}
```

> `@Qualifier`가 `@Primary`보다 우선순위 높음 — 좁고 구체적인 설정이 이김

## 빈 스코프 (Bean Scope)

| 스코프 | 동작 | 사용 시점 |
|--------|------|----------|
| **싱글톤** (기본값) | 컨테이너에 딱 1개, 모두 공유 | 상태 없는 비즈니스 로직, DataSource |
| **프로토타입** | 요청할 때마다 새 객체 생성 | 매번 독립 객체가 필요할 때 |

```java
@Component
@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
public class GasolineCar implements Car { }
```

> 프로토타입은 컨테이너가 생성/주입/초기화까지만 관리 — 소멸은 개발자가 직접 처리

## 빈 생명주기와 콜백

스프링 빈이 생성되고 사라지는 순서:

```
컨테이너 기동 → 빈 생성 → 의존관계 주입 → @PostConstruct → 비즈니스 로직 → @PreDestroy → 컨테이너 종료
```

생명주기 콜백을 구현하는 방법은 3가지인데, 오늘 2가지를 실습했음.

**방법 1: 어노테이션 방식 (권장)** — `TempFileSupport.java`
```java
// @Component  ← 실습 중 주석 처리하고 TempFileSupport2로 전환
public class TempFileSupport {
    @Value("resources/temp.log")
    private String filePath;

    public TempFileSupport() {
        System.out.println("생성자 호출됨. " + filePath); // filePath = null!
    }

    @PostConstruct  // 의존 주입 완료 후 자동 실행
    public void init() {
        System.out.println(filePath + " 경로의 FileOutputStream 생성.");
    }

    @PreDestroy     // 컨테이너 종료 직전 자동 실행
    public void close() {
        System.out.println(filePath + " 경로의 FileOutputStream 닫기.");
    }
}
```

**방법 2: 인터페이스 구현 방식 (스프링 종속 — 비권장)** — `TempFileSupport2.java`
```java
@Component
public class TempFileSupport2 implements InitializingBean, DisposableBean {
    @Value("resources/temp.log")
    private String filePath;

    @Override
    public void afterPropertiesSet() throws Exception {  // @PostConstruct 역할
        System.out.println(filePath + " 경로의 FileOutputStream 생성.");
    }

    @Override
    public void destroy() throws Exception {  // @PreDestroy 역할
        System.out.println(filePath + " 경로의 FileOutputStream 닫기.");
    }
}
```

> 콘솔에서 생성자 시점엔 `filePath = null` 찍힘 — 주입 전이라 당연한 것!  
> 인터페이스 방식은 스프링에 종속되므로 어노테이션 방식(`@PostConstruct`/`@PreDestroy`)을 권장

## 스프링 부트 입문

### 스프링 프레임워크 vs 스프링 부트

| | 스프링 프레임워크 | 스프링 부트 |
|--|-----------------|------------|
| 라이브러리 관리 | 버전을 직접 하나씩 맞춰야 함 | 스타터가 호환 버전 세트를 한 번에 제공 |
| WAS | 외부 Tomcat 별도 설치 + WAR 배포 | Tomcat 내장 + JAR로 단독 실행 |
| 설정 | XML/Java 설정 직접 작성 | 자동 설정 (Auto Configuration) |

### 프로젝트 생성 방법 2가지

1. **Spring Initializr** (`https://start.spring.io`) — 웹에서 설정하고 zip 다운로드
2. **IntelliJ에서 직접** — File → New → Project → Spring Boot 선택

### build.gradle 핵심 키워드

```groovy
dependencies {
    implementation '...'        // 컴파일 + 런타임 (일반 라이브러리)
    compileOnly '...'           // 컴파일만 (런타임 제외 — lombok 등)
    annotationProcessor '...'   // 어노테이션 처리 전용 컴파일러 (lombok)
    runtimeOnly '...'           // 런타임만 (DB 드라이버 등)
    testImplementation '...'    // 테스트 컴파일 + 런타임
    testRuntimeOnly '...'       // 테스트 런타임만 (JUnit 런처)
}
```

### 주요 스타터 라이브러리

| 스타터 | 역할 |
|--------|------|
| `spring-boot-starter` | 기본 스타터 (DI 컨테이너, 로깅) |
| `spring-boot-starter-web` | 내장 Tomcat + Spring MVC |
| `spring-boot-starter-test` | JUnit 5, Mockito, AssertJ |
| `spring-boot-starter-data-jpa` | JPA (ORM) |
| `lombok` | getter/setter 자동 생성 |
| `mysql-connector-j` | MySQL 드라이버 |

## 배운 점

1. ✅ `@ComponentScan` — 패키지 하위를 자동 탐색, `@Component` 붙은 클래스를 빈으로 등록
2. ✅ `@Autowired` — 타입 기준으로 컨테이너에서 자동 주입 (생성자/수정자/필드 3가지 방식)
3. ✅ 동일 타입 빈 충돌 시 `@Primary`(기본값) 또는 `@Qualifier`(직접 지정)로 해결
4. ✅ 싱글톤은 하나 공유, 프로토타입은 매번 새로 생성
5. ✅ `@PostConstruct` / `@PreDestroy` — 생성자 말고 초기화/소멸 콜백에서 외부 리소스 처리
6. ✅ 스프링 부트 = 스프링 + 자동설정 + 내장 WAS + 스타터 의존성
7. ✅ Spring Initializr로 프로젝트 뼈대를 빠르게 생성 가능
8. ✅ `build.gradle`의 `implementation` / `compileOnly` / `runtimeOnly` 차이

## 오늘 핵심 정리

```
컴포넌트 스캔
  @ComponentScan → @Component 자동 탐색 → 빈 등록
  빈 이름 = 클래스명 첫 글자 소문자

자동 주입
  @Autowired → 타입 기준 자동 주입
  충돌 → @Primary(기본) or @Qualifier("빈이름")(직접 지정)
  우선순위: @Qualifier > @Primary

빈 스코프
  싱글톤(기본): 1개 공유
  프로토타입: 요청마다 새 객체

생명주기
  생성 → 주입 → @PostConstruct → 로직 → @PreDestroy → 종료

스프링 부트
  스타터 → 버전 충돌 없이 한 번에
  내장 Tomcat → JAR 단독 실행
  Spring Initializr → 프로젝트 뼈대 자동 생성
```
