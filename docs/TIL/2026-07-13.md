## HTTP 프로토콜

브라우저(클라이언트)와 서버가 데이터를 주고받는 규칙.

### 요청(Request) 메시지 구조

```
GET /board/list.html HTTP/1.1       ← 요청 라인 (메서드 + URL + 버전)
Host: localhost:8080                 ← 헤더들
Accept: text/html
User-Agent: Mozilla/5.0 ...
                                     ← 빈 줄 (헤더 끝)
(GET은 바디 없음)
```

POST는 바디에 데이터가 있음:
```
POST /board/write.html HTTP/1.1
Content-Type: application/x-www-form-urlencoded

title=hello&content=hi&author=namu  ← 바디 (폼 데이터)
```

### 응답(Response) 메시지 구조

```
HTTP/1.1 200 OK                      ← 상태 라인
Content-Type: text/html              ← 헤더들
Date: Mon, 13 Jul 2026 01:56:52 GMT

<!DOCTYPE html>...                   ← 바디 (HTML)
```

### 주요 상태 코드

| 코드 | 의미 |
|------|------|
| 200 | 성공 |
| 201 | 생성됨 (POST) |
| 301/302 | 리다이렉트 (다른 URL로 이동) |
| 304 | 변경 없음 (캐시 사용) |
| 400 | 잘못된 요청 |
| 401 | 로그인 필요 |
| 403 | 권한 없음 |
| 404 | 찾을 수 없음 |
| 500 | 서버 에러 |

### HTTP 주요 메서드

| 메서드 | 용도 | 예시 |
|--------|------|------|
| GET | 조회 | 게시글 목록, 상세 조회 |
| POST | 등록 | 게시글 작성, 회원가입 |
| PUT | 전체 수정 | 게시글 전체 항목 수정 |
| PATCH | 일부 수정 | 게시글 제목만 수정 |
| DELETE | 삭제 | 게시글 삭제 |

### HTTP 두 가지 특징

- **비연결성(Connectionless)**: 응답하면 연결 끊음. 서버가 클라이언트를 기억 안 함
- **무상태(Stateless)**: 서버는 이전 요청을 기억 안 함 → 로그인 유지를 위해 Cookie/Session/JWT 필요

## 3티어 아키텍처

웹 시스템을 3개 영역으로 나눠서 운영하는 구조:

```
브라우저 (Presentation)    ← HTML/CSS/JS 보여주는 곳
    ↕ HTTP
스프링 부트 서버 (Application)  ← 비즈니스 로직 처리
    ↕ SQL
데이터베이스 (Data)         ← 데이터 저장
```

물리적으로 서버를 분리해서 특정 영역에 부하가 몰리면 그 서버만 늘리면 됨.

## MVC 패턴

스프링 서버 안에서 역할을 3가지로 나누는 설계 방식:

```
클라이언트 요청
    ↓
Controller  ← 요청 받아서 Service 호출, 결과를 View에 전달 (홀 직원)
    ↓
Service     ← 비즈니스 로직 수행 (주방 요리사)
    ↓
Repository  ← DB에서 데이터 꺼내옴 (창고)
    ↓
View        ← HTML로 화면 만들어서 응답 (메뉴판/플레이팅)
```

### 계층 간 데이터 전달 객체

| 이름 | 역할 |
|------|------|
| **DTO** | 계층 간 데이터 전달용 (getter/setter만 있음) |
| **Entity** | DB 테이블과 1:1 대응하는 객체 |
| **DAO** | DB에 직접 SQL 실행하는 객체 |

## 스프링 컨트롤러 만들기

### 핵심 어노테이션

```java
@Controller          // 이 클래스가 웹 요청 처리 담당
@RequestMapping      // URL 경로 매핑
@GetMapping          // GET 요청 매핑
@PostMapping         // POST 요청 매핑
@ResponseBody        // 반환값을 HTTP 바디에 직접 씀 (View 거치지 않음)
```

### 실습 코드 — `BoardController.java`

```java
@Controller
public class BoardController {

    // 게시글 더미 데이터 (DB 없을 때 임시로 List에 저장)
    private final List<PostDto> fakePosts;

    public BoardController() {
        fakePosts = new ArrayList<>();
        PostDto post1 = new PostDto();
        post1.setId(1);
        post1.setTitle("1번 게시글");
        post1.setAuthor("하루");
        post1.setCreatedAt(LocalDateTime.now());
        fakePosts.add(post1);
        // ...
    }

    // GET /01/board/list.html → 게시글 목록 HTML 직접 반환
    @GetMapping("/01/board/list.html")
    @ResponseBody
    public String getBoardList() {
        List<PostDto> posts = getPosts();

        String result = "<table>...";
        for (PostDto post : posts) {
            result += "<tr><td>%s</td><td>%s</td>...</tr>"
                .formatted(post.getId(), post.getTitle(), ...);
        }
        result += "</table>";
        return result;
    }
}
```

### DTO 실습 코드 — `PostDto.java`

```java
public class PostDto {
    private int id;
    private String title;
    private String content;
    private String author;
    private LocalDateTime createdAt;

    // getter / setter
    public int getId() { return id; }
    public void setId(int id) { this.id = id; }
    // ...
}
```

게시글 하나의 데이터를 담는 그릇. DB 없이 테스트할 때 이 DTO를 List에 담아서 씀.

### Forward vs Redirect

```java
// Forward: 서버 내부에서 뷰로 넘김 → URL 안 바뀜
return "board/list";  // templates/board/list.html 찾아서 렌더링

// Redirect: 브라우저한테 "이 URL로 다시 요청해" → URL 바뀜
return "redirect:/board/list";  // 브라우저가 GET /board/list 재요청
```

> POST 후 목록으로 갈 때 Redirect 씀 → 새로고침해도 게시글 이중 등록 안 됨

## 배운 점

1. ✅ HTTP 요청/응답은 텍스트 기반 — 요청 라인 + 헤더 + 빈 줄 + 바디 구조
2. ✅ GET은 URL에 데이터, POST는 바디에 데이터
3. ✅ 상태 코드 — 2xx 성공, 3xx 리다이렉트, 4xx 클라이언트 잘못, 5xx 서버 잘못
4. ✅ 3티어 = 브라우저 / 스프링 서버 / DB 물리적으로 분리
5. ✅ MVC = Controller(요청 처리) / Service(로직) / View(화면) 역할 분리
6. ✅ DTO = 계층 간 데이터 전달 객체, getter/setter만 있음
7. ✅ `@Controller` + `@GetMapping` + `@ResponseBody`로 URL에 HTML 직접 반환
8. ✅ DB 없을 때 `List<PostDto>`로 가짜 데이터 만들어서 테스트 가능
9. ✅ Redirect는 브라우저가 새 URL로 재요청 → 새로고침 이중 등록 방지

## 오늘 핵심 정리

```
HTTP
  요청: 메서드 + URL + 헤더 + 바디
  응답: 상태코드 + 헤더 + 바디
  GET → 조회 (바디 없음)
  POST → 등록 (바디에 데이터)

MVC 흐름
  클라이언트 → Controller → Service → Repository → DB
  DB → Repository → Service → Controller → View → 클라이언트

컨트롤러
  @Controller + @GetMapping("/url") + @ResponseBody
  → URL 매핑해서 HTML 문자열 반환

DTO
  PostDto { id, title, content, author, createdAt }
  getter/setter만 있는 데이터 전달 전용 객체

Forward vs Redirect
  Forward  → URL 안 바뀜 (서버 내부 이동)
  Redirect → URL 바뀜 (브라우저 재요청)
```
