### 1. 순수 JDBC의 문제점

DB 연동을 직접 구현하면 아래처럼 반복되는 코드가 굉장히 많다.

```java
// PureJdbcPostRepository.java 중 findAll()
public List<PostDto> findAll() {
    String sql = "SELECT id, title, author, secret, created_at AS createdAt FROM post2";

    Connection conn = null;
    Statement stmt = null;
    ResultSet rs = null;
    List<PostDto> result = new ArrayList<>();

    try {
        // 1. DB 연결
        conn = DriverManager.getConnection(url, user, password);
        // 2. SQL 실행 객체 생성
        stmt = conn.createStatement();
        // 3. SQL 실행
        rs = stmt.executeQuery(sql);
        // 4. 결과 처리
        while (rs.next()) {
            PostDto postDto = new PostDto();
            postDto.setId(rs.getInt("id"));
            postDto.setTitle(rs.getString("title"));
            // ...
            result.add(postDto);
        }
    } catch (Exception e) {
        System.out.println("에러 발생: " + e.getMessage());
    } finally {
        // 5. 리소스 해제 (생성 역순)
        try { if (rs != null) rs.close(); } catch (Exception e) {}
        try { if (stmt != null) stmt.close(); } catch (Exception e) {}
        try { if (conn != null) conn.close(); } catch (Exception e) {}
    }
    return result;
}
```

**문제점 2가지**

1. **보일러플레이트 코드**: 커넥션 연결 → SQL 실행 → ResultSet 처리 → 리소스 해제가 모든 메서드마다 반복됨
2. **예외 처리 강제**: `java.sql.SQLException`은 체크 예외라 Service 계층까지 `throws SQLException`이 퍼짐 → Service가 DB 기술에 종속됨

### 2. JdbcTemplate으로 해결

스프링이 반복되는 흐름(커넥션 획득 → 실행 → 리소스 반납)을 **템플릿**으로 대행하고, SQL과 매핑 로직만 개발자가 작성하면 된다.

#### 2-1. 의존성 설정

```gradle
// build.gradle
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-jdbc'
    runtimeOnly 'com.mysql:mysql-connector-j'
}
```

#### 2-2. DB 접속 정보 (`application.properties`)

```properties
spring.datasource.url=jdbc:mysql://localhost:3306/board_db?serverTimezone=UTC&useSSL=false&allowPublicKeyRetrieval=true
spring.datasource.username=root
spring.datasource.password=root

# 구동 시 schema.sql, data.sql 자동 실행
spring.sql.init.mode=always
```

스프링 부트가 `spring.datasource.*` 표준 키로 **DataSource와 JdbcTemplate 빈을 자동 생성**해준다.

#### 2-3. DB 초기화 스크립트

```sql
-- schema.sql (테이블 생성)
CREATE TABLE IF NOT EXISTS post2 (
    id INT AUTO_INCREMENT PRIMARY KEY,
    author VARCHAR(10) NOT NULL,
    title VARCHAR(200) NOT NULL,
    content TEXT NOT NULL,
    secret BOOLEAN NOT NULL DEFAULT FALSE,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

```sql
-- data.sql (초기 데이터 삽입)
INSERT INTO post2 (secret, title, content, created_at, author) VALUES
    (TRUE, '첫번째 게시글', '오늘도 자바 공부를 열심히 하고 있습니다.', CURRENT_TIMESTAMP, '하루'),
    (FALSE, '두번째 게시글', '오늘도 자바 공부를 열심히 하고 있습니다.', CURRENT_TIMESTAMP, '나무');
```

`spring.sql.init.mode=always`로 설정하면 앱 시작할 때마다 두 파일이 순서대로 실행된다.

### 3. RowMapper - ResultSet → 자바 객체 변환

DB 조회 결과(`ResultSet`)의 각 행(Row)을 자바 객체로 변환하는 규칙을 정의하는 인터페이스.

```java
@FunctionalInterface
public interface RowMapper<T> {
    T mapRow(ResultSet rs, int rowNum) throws SQLException;
}
```

함수형 인터페이스이기 때문에 **람다식**으로 간결하게 작성 가능하다.

```java
// JdbcTemplatePostRepository.java
private final RowMapper<PostDto> postRowMapper = (ResultSet rs, int rowNum) -> {
    return PostDto.builder()
            .id(rs.getInt("id"))
            .title(rs.getString("title"))
            .author(rs.getString("author"))
            .content(rs.getString("content"))
            .secret(rs.getBoolean("secret"))
            .createdAt(rs.getObject("created_at", LocalDateTime.class))
            .build();
};
```

### 4. JdbcTemplate CRUD 구현

```java
// JdbcTemplatePostRepository.java
@Repository
public class JdbcTemplatePostRepository implements PostRepository {

    private final JdbcTemplate jdbcTemplate;

    public JdbcTemplatePostRepository(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }

    // 다건 조회
    @Override
    public List<PostDto> findAll() {
        return jdbcTemplate.query("SELECT * FROM post2", postRowMapper);
    }

    // 단건 조회
    @Override
    public PostDto findById(int id) {
        return jdbcTemplate.queryForObject(
            "SELECT * FROM post2 WHERE id = ?", postRowMapper, id);
    }

    // 등록
    @Override
    public void save(PostDto post) {
        jdbcTemplate.update(
            "INSERT INTO post2 (title, author, content) VALUES (?, ?, ?)",
            post.getTitle(), post.getAuthor(), post.getContent());
    }

    // 수정
    @Override
    public void update(PostDto post) {
        jdbcTemplate.update(
            "UPDATE post2 SET title = ?, author = ?, content = ? WHERE id = ?",
            post.getTitle(), post.getAuthor(), post.getContent(), post.getId());
    }

    // 삭제
    @Override
    public void deleteById(int id) {
        jdbcTemplate.update("DELETE FROM post2 WHERE id = ?", id);
    }
}
```

**핵심 메서드 정리**

| 메서드 | 용도 | 반환값 |
|--------|------|--------|
| `query(sql, rowMapper)` | 다건 조회 (목록) | `List<T>` |
| `queryForObject(sql, rowMapper, args)` | 단건 조회 | `T` (없으면 예외) |
| `update(sql, args...)` | INSERT / UPDATE / DELETE | 반영된 행 수(int) |

**`?` 파라미터**: SQL에 값을 직접 문자열로 붙이면 SQL 인젝션 위험이 있어서, `?`로 표시하고 뒤에 순서대로 값을 넘긴다.

### 5. 인터페이스 덕분에 구현체 교체가 쉬움

오늘 실습의 핵심 포인트. `PostRepository` 인터페이스를 구현한 클래스가 3개다:

```
PostRepository (인터페이스)
├── MemoryPostRepository  - 메모리에 List로 저장
├── PureJdbcPostRepository - 순수 JDBC로 MySQL 연동
└── JdbcTemplatePostRepository - JdbcTemplate으로 MySQL 연동
```

`PostServiceImpl`은 `PostRepository` 타입만 알고 있기 때문에, `@Repository`가 달린 구현체를 교체해도 Service/Controller 코드는 **전혀 수정할 필요가 없다**.

---

### 6. @Value로 설정값 주입

```java
@Repository
public class PureJdbcPostRepository implements PostRepository {

    @Value("${spring.datasource.url}")
    private String url;

    @Value("${spring.datasource.username}")
    private String user;

    @Value("${spring.datasource.password}")
    private String password;
}
```

`application.properties`에 있는 값을 `@Value("${키이름}")`으로 필드에 주입받을 수 있다.  
코드에 DB 주소/비번을 하드코딩하지 않아도 된다.

### 7. Spring Data JDBC 개요 (맛보기)

JdbcTemplate은 SQL을 직접 써야 하는데, **Spring Data JDBC**는 단순 CRUD SQL을 프레임워크가 자동 생성해준다.

```gradle
// build.gradle
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-data-jdbc'
    // starter-data-jdbc가 starter-jdbc를 포함하므로 JdbcTemplate도 같이 사용 가능
}
```

JPA의 복잡한 지연 로딩, 영속성 컨텍스트 없이 직관적으로 동작한다.

## 정리

| 단계 | 기술 | 특징 |
|------|------|------|
| 1단계 | 메모리(List) | 앱 재시작하면 데이터 사라짐 |
| 2단계 | 순수 JDBC | DB 연동 O, 코드가 너무 길고 반복적 |
| 3단계 | JdbcTemplate | DB 연동 O, SQL만 작성하면 됨 (현재) |
| 4단계 | Spring Data JDBC | CRUD SQL도 자동 생성 (다음 주) |

인터페이스로 설계해뒀기 때문에 단계가 올라갈수록 구현체만 교체하면 된다.
