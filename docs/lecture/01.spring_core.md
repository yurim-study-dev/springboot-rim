# 스프링 프레임워크 핵심 원리

## 학습 목표
- 프레임워크와 라이브러리의 개념 및 제어의 역전(IoC) 원리를 구분함.
- 빌드 도구인 Gradle과 Maven의 구조를 분석함.
- 스프링 프레임워크 핵심 설계 원칙인 제어의 역전(IoC), 의존성 주입(DI), 관점 지향 프로그래밍(AOP)의 탄생 배경과 구동 원리를 이해함.
- 스프링 빈의 생성과 소멸에 이르는 전체 생명주기와 이를 제어하는 스프링 컨테이너의 관리 절차를 파악함.

- 기술 교체 시에도 비즈니스 코드를 보존하는 서비스 추상화(PSA)의 개념적 유연성을 이해함.

# 1. 스프링 입문

## 1.1 프레임워크와 라이브러리

### 1.1.1 라이브러리(Library)
- 애플리케이션 개발에 필요한 특정 기능들이 미리 구현된 재사용 가능한 코드, 클래스, 함수의 집합.
- 날짜 계산이나 데이터 암호화처럼 복잡하고 반복적인 기능 코드를 매번 직접 설계하지 않고, 이미 품질과 신뢰성이 보장된 리소스를 활용하여 전체 결함을 낮추고 개발 생산성을 높이기 위해 필요함.
- 개발자가 전체적인 프로그램의 실행 흐름을 직접 제어하며, 필요한 시점에 적절한 라이브러리를 주도적으로 호출하여 사용함.
- 칼과 도마, 냄비만 사용하여 요리하던 환경(자바의 표준 API만 사용)에서 필요에 따라 믹서기, 오븐, 전자레인지처럼 요리를 단축해 주는 조리 도구(라이브러리)를 이용해서 직접 조리하는 환경.
- 냉동실에서 고기를 꺼내서 자연 해동을 하든, 전자레인지의 해동 버튼을 누르든, 요리의 주체인 개발자가 원하는 도구를 직접 선택하고 활용하는 것처럼, 코드의 실행 흐름과 제어 권한이 개발자에게 있음.

### 1.1.2 프레임워크(Framework)
- 애플리케이션을 완성하는 데 필요한 전체적인 설계 구조와 구동 규칙을 기본 뼈대로 미리 만들어 제공하고, 개발자가 세부적인 기능 코드를 약속된 위치에 끼워 넣으면 프로그램이 가동될 때 적절한 타이밍에 개발자의 코드를 호출하여 전체 실행 흐름을 통제해 주는 소프트웨어 개발의 기반이 되는 플랫폼.
- 다수의 작업자가 참여하는 거대 프로젝트에서 프로그램의 기본 뼈대와 규칙을 강제함으로써, 개발자 개개인의 설계 차이에서 오는 품질 격차를 최소화하고 코드를 일관되게 정립하기 위해 필요함.
- 데이터 접근이나 사용자 인증 등 매번 공통 설계가 필요한 작업은 프레임워크에 위임하여 생산 효율을 높이고 개발자는 핵심 비즈니스 구현에만 집중할 수 있음.
- 개발자는 프레임워크가 정의한 설계 규칙에 따라 코드를 작성하여 삽입하며, 프레임워크가 이 코드를 적절한 타이밍에 호출하여 실행함.
- 전문 요리사를 고용한 사람이 식재료를 약속된 위치에 준비해 두기만 하면, 요리사는 본인이 사용할 전용 조리도구까지 스스로 챙겨서 요리 중 필요한 시점에 재료를 가져가서 조리하는 흐름과 유사함.
- 전체 요리 진행 순서와 조리도구 세팅 및 관리 등 작동 흐름은 요리사가 관장하며, 개발자는 지정된 장소에 원하는 식재료를 채워두는 역할에만 집중함.
    - 제어의 역전 (Inversion of Control)
        - 개발자가 직접 객체를 생성하고 코드의 실행 흐름을 주도하는 것이 아니라, 프레임워크가 객체의 생명주기를 관리하고 작성된 코드를 알맞은 타이밍에 실행하여 흐름을 통제하는 소프트웨어 설계 원칙.
        - 스프링 프레임워크에서는 스프링 컨테이너가 객체의 생성 및 소멸, 관계 설정 처리를 대행하여, 개발자가 직접 흐름을 통제하지 않고 핵심 비즈니스 로직[^1] 설계에만 전념할 수 있도록 지원함.
        - 영화배우 오디션에서 담당자들이 지원자에게 자주 언급하는 헐리우드 원칙("Don't call us, we'll call you.")에서 유래한 소프트웨어 설계 규칙.
        - 개발자가 프레임워크의 코드를 직접 호출하여 가동하지 않고, 정해진 규칙에 따라 코드를 등록해 두면 프레임워크가 필요할 때 해당 코드를 호출함.

## 1.2 스프링 프레임워크 개요
- 자바 엔터프라이즈 애플리케이션 개발을 편리하게 해주는 오픈소스 프레임워크.
- 주요 장점:
    - 데이터베이스 연결이나 사용자 인증처럼 번거롭고 복잡한 기본 설정 및 인프라 구축 작업을 대신 해결해 주어, 개발자가 핵심 비즈니스 로직[^1]<sup>2</sup> 개발에만 집중하도록 지원.

    - 필요한 기능들이 부품처럼 미리 규격화되어 있어서, 약속된 규칙에 맞춰 조립하듯이 빠르게 대규모 웹 서비스 시스템 신속 구축.
    - 데이터베이스 종류나 연동하는 외부 시스템의 기술 규격이 도중에 바뀌더라도, 이미 작성해 둔 프로그램 코드를 고치지 않고 부품만 유연하게 교체.
- 주요 단점:
    - 제공하는 기술 범위가 광범위하고 핵심 개념이 추상적이어서, 초보 개발자가 구동 원리를 완벽히 파악하기 전까지 학습 장벽이 존재함.
    - 프로젝트를 시작할 때 수많은 구성 파일이나 라이브러리 설정을 수동으로 정밀하게 제어해야 하므로, 초기 환경을 구축하는 절차가 매우 복잡함.
- 과거 자바 진영의 기술 규격이었던 EJB(Enterprise JavaBeans)는 보안과 분산 처리 기능을 제공했으나, 간단한 기능 하나를 만들 때도 수많은 설정 파일 작성과 무거운 인터페이스 상속을 강제하여 개발 생산성이 저하되는 원인으로 지목.
- 스프링은 기술 종속성 없이 가볍고 순수한 자바 기본 객체인 POJO(Plain Old Java Object)만을 사용하여 유연한 객체지향 설계 지원.

## 1.3 스프링 핵심 용어 사전
- 의존성 (Dependency)
    - 한 객체가 자신의 역할을 수행하는 과정에서 다른 객체를 호출하거나 참조하여 작동을 의지하는 협력 관계
    - 예를 들어 운전자(Driver) 객체가 정상 동작하기 위해 자동차(Car) 객체를 필요로 하는 것처럼, 하나의 컴포넌트가 작동하기 위해 필수적으로 결합되어야 하는 대상 객체나 리소스
- 의존성 주입 (DI, Dependency Injection)
    - 어떤 객체가 필요로 하는 협력 객체를 스스로 직접 생성하여 확보하지 않고, 외부의 관리 장치(컨테이너)로부터 주입(전달)받아 사용하는 설계 기법
    - 객체 간 결합도를 낮추고 모듈 독립성을 높여 기술의 사후 변경을 용이하게 유도
- 스프링 빈 (Spring Bean)
    - 스프링 컨테이너가 스스로 생성, 관계 조립, 생명주기 관리를 전담하는 대상 자바 객체
    - 개발자가 직접 new 연산자로 선언하여 사용하는 일반 자바 객체와 구별되며, 컨테이너에 의해 제어권이 위임된 컴포넌트만을 지칭
- 스프링 컨테이너 (Spring Container / IoC Container)
    - 스프링 빈의 인스턴스화, 구성설정, 생명주기 제어 및 의존 관계 연결 처리를 담당하는 핵심 구동 엔진
    - 개발자가 작성한 컴포넌트들을 유기적으로 조립하는 자동화 공장 역할을 수행
- 설정 메타데이터 (Configuration Metadata)
    - 스프링 컨테이너가 어떤 객체를 생성하고 어떻게 의존 관계를 이어 붙여 조립할지 지시하는 조립 설계도 문서
    - 과거에는 XML 문서 형태로 기재했으나, 현대 스프링 표준에서는 자바 설정 클래스 파일 상단에 Configuration, Bean 등의 어노테이션 지시어를 부여하여 관리

# 2. 빌드 도구 (Build Tool)

## 2.1 스프링 프로젝트의 빌드 및 배포 흐름

### 2.1.1 빌드와 배포
- 빌드(Build)의 정의
    + 개발자가 작성한 자바 소스코드를 컴퓨터가 실행할 수 있는 최종 소프트웨어 산출물(.jar)로 변환하는 일련의 작업 과정
- 빌드의 세부 과정
    + 1. 의존성 해결(Dependency Resolution): 빌드 설정 파일(build.gradle 등)을 파싱[^2]하여 프로젝트 가동에 필요한 외부 라이브러리들을 원격 저장소에서 다운로드
    + 2. 컴파일(Compile): 사람이 읽기 쉬운 자바 소스코드(.java)를 자바 가상 머신(JVM)이 실행할 수 있는 바이트코드(.class)로 번역함
    + 3. 테스트 실행(Test): JUnit 등의 테스트 프레임워크를 가동하여 작성된 비즈니스 로직의 결함 유무를 빌드 완료 전에 최종 검증함
    + 4. 패키징(Packaging): 컴파일을 마친 바이트코드와 리소스 파일(properties, yml 등)을 단일 실행 가능한 형태(.jar)로 압축함
- 배포(Deploy)의 정의
    + 빌드를 통해 생성된 최종 소프트웨어 산출물(.jar)을 실제 서비스를 제공할 운영 서버 환경에 올리고 실행하여 사용자가 서비스를 이용할 수 있도록 만드는 과정

### 2.1.2 스프링 프로젝트의 빌드와 배포 절차
- 소스코드 작성: 개발자가 로컬 PC에서 요구사항에 맞는 자바 프로그램 소스코드와 애플리케이션 설정 파일을 작성함
- 빌드 도구 가동: Gradle 또는 Maven 빌드 도구를 실행하여 작성된 소스코드를 컴파일하고 오류 검증용 테스트를 통과시킨 뒤, 최종 실행 파일(.jar)로 압축 패키징함
- 실행 파일(jar) 전송: 로컬 PC 빌드로 산출된 jar 실행 파일을 실제 서비스를 운영할 원격 서버나 클라우드 서버 환경으로 전송함
- 배포 및 service 실행: 전송받은 서버 환경에서 java -jar 명령어로 애플리케이션을 단독 구동하여 사용자의 실제 요청을 처리할 수 있도록 서비스를 개시함

## 2.2 Maven과 Gradle
- 외부 라이브러리를 다운로드하고 소스코드를 컴파일하여 최종 실행 파일로 패키징하는 빌드 자동화 도구.

### 2.2.1 Maven (메이븐)
#### 1) 개요
- 아파치 재단[^3]에서 개발한 자바 프로젝트 빌드 도구로, XML[^4] (pom.xml) 기반의 정형화된 라이프사이클 모델을 따름.

#### 2) 특징
- 프로젝트 설정과 의존성[^5] 정보를 XML 형태로 엄격하게 표현하며, 설정 방식이 규격화되어 일관된 구조로 관리할 수 있음.

#### 3) 한계
- 프로젝트 규모가 커질수록 XML 구조가 비대해지고 태그 중복이 증가하여 가독성이 저하되며, 빌드 스크립트[^6] 내 동적 논리 제어가 어려움.

### 2.2.2 Gradle (그레이들)
#### 1) 개요
- Groovy 또는 Kotlin DSL(Domain Specific Language) 설정 언어를 탑재하여, 메이븐의 선언적 설정[^7] 장점과 프로그래밍 유연성을 결합한 표준 빌드 도구.

#### 2) 특징
- 단순 설정을 넘어 빌드 스크립트[^6] 내에서 변수 선언 및 제어문 활용 등 동적인 빌드 스크립트 작성 지원.



#### 3) 성능적 장점
- 이전 빌드 결과물을 재사용하고, 변경이 발생한 소스 파일만 선별적으로 컴파일하는 증분 빌드 기법을 적용하여 빌드 속도를 개선.

### 2.2.3 의존성 관리 (Dependency Management)
- 설정 파일에 라이브러리 명칭만 명시하면 빌드 도구가 네트워크를 통해 필요한 라이브러리 자동 다운로드.

그레이들 기반 자바 프로젝트를 생성하면 다음과 같은 표준 build.gradle 빌드 설정 파일이 자동으로 생성됨.
- build.gradle
```groovy
plugins {
    id 'java'
}

group = 'net.likelion.bebc25'
version = '1.0-SNAPSHOT'

repositories {
    mavenCentral()
}

dependencies {
    testImplementation platform('org.junit:junit-bom:6.0.0')

}
```

- plugins: 프로젝트 빌드 프로세스에 필요한 추가 기능과 도구를 그레이들에 적용하는 영역으로, id 'java'는 그레이들이 자바 프로젝트를 인식하여 컴파일 및 빌드할 수 있는 기본 기능을 제공
- group, version: 빌드된 산출물(JAR 등)을 공유하거나 배포할 때 구분하기 위한 고유 식별 정보와 버전을 명시
- repositories: 프로젝트에 필요한 외부 라이브러리들을 받아올 서버 저장소를 지정하며, mavenCentral()은 메이븐 중앙 라이브러리 저장소를 지칭
- dependencies: 프로젝트 컴파일 및 테스트 구동에 사용할 외부 라이브러리들을 그레이들이 다운로드하도록 선언하는 영역으로, 기본적으로 단위 테스트를 위한 junit-jupiter 라이브러리가 명시되어 있음
- test: 테스트 실행 환경을 지정하는 태스크 설정 영역으로, useJUnitPlatform()을 설정하여 JUnit 5 테스트 엔진으로 테스트가 실행되도록 지정

## 3.4 스프링 프레임워크 의존성 추가
순수 자바 그레이들 프로젝트에서 스프링의 핵심 원리인 IoC와 DI를 가동하기 위해서는 스프링 프레임워크 라이브러리를 build.gradle 설정에 수동으로 선언해 주어야 함.

### 3.4.1 spring-context 의존성 선언

- build.gradle
  ```groovy
  dependencies {

  }
  ```

### 3.4.2 의존성 로드 및 빌드 갱신
의존성을 반영하여 외부 라이브러리를 프로젝트에 적용하려면 그레이들 동기화(Sync) 작업을 수행해야 함.
- 방법 1 (편집창 단추 이용): build.gradle 파일을 수정한 뒤 자바 편집창 우측 상단 영역에 일시적으로 노출되는 파란색 새로고침 기호가 표시된 작은 코끼리 모양의 Gradle 변경 사항 로드(Sync Gradle Changes) 단추를 클릭
- 방법 2 (Gradle 탭 이용): 인텔리제이 화면 우측 가장자리 세로 툴바에서 코끼리 아이콘(Gradle)을 클릭하여 Gradle 탭 창을 활성화한 뒤, 탭 창 내부의 도구 모음 맨 좌측에 위치한 회전 화살표(새로고침) 모양의 Sync All Gradle Projects 단추를 클릭
- 다운로드 확인: 그레이들이 메이븐 중앙 저장소로부터 spring-context 라이브러리와 이에 연동되는 스프링 코어 모듈들을 자동으로 다운로드하여 컴파일 클래스 패스에 정상 적재하는지 최종 확인
    - Project 창의 SpringCore 프로젝트 아래에 External Libraries를 확장해서 `Gradle: org.springframework:spring-xxx:7.0.8` 라이브러리 들이 추가 되었는지 확인

### 4.1.2 의존성을 느슨하게 결합하는 방식
```java
interface Car {
  void startEngine();
  void drive();
  void stopEngine();
}

class GasolineCar implements Car {
  @Override
  public void startEngine() {
    System.out.println("가솔린 엔진에 점화하여 시동을 켭니다.");
  }
  @Override
  public void drive() {
    System.out.println("가솔린 동력으로 주행합니다.");
  }
  @Override
  public void stopEngine() {
    System.out.println("가솔린 공급을 차단하여 시동을 끕니다.");
  }
}

class HybridCar implements Car {
  @Override
  public void startEngine() {
    System.out.println("시스템 전원을 인가하고 하이브리드 시동을 켭니다.");
  }
  @Override
  public void drive() {
    System.out.println("전기와 엔진을 혼합하여 효율적으로 주행합니다.");
  }
  @Override
  public void stopEngine() {
    System.out.println("하이브리드 시스템 종료 처리를 작동합니다.");
  }
}

class Driver {
  private Car car; // 인터페이스에 의존 (느슨한 결합)

  // 생성자를 통해 외부에서 Car 구현체를 주입 받음
  public Driver(Car car) {
    this.car = car;
  }

  public void driveCar() {
    car.startEngine();
    car.drive();
    car.stopEngine();
  }
}
```
- 새로운 클래스(예: ElectricCar)를 정의하여 구현하고, 외부에서 주입해 주는 객체만 바꿔 끼워주면 운전자 코드는 그대로 재사용함

### 4.1.3 제어의 역전 (IoC, Inversion of Control)
##### 1) 제어 주도권의 양도
- [직접 결합 방식](#411-직접-결합-방식)에서는 개발자가 Driver 클래스 내부에서 직접 new GasolineCar() 또는 new HybridCar()를 기재하여 협력 객체의 생성과 의존성 조립을 스스로 통제함
- [느슨한 결합 방식](#412-느슨한-결합-방식)을 적용하면 Driver 클래스는 단지 Car 인터페이스에만 의존하며, 구체적인 구현 객체의 생성과 주입 권한은 외부로 넘어감
- 이처럼 프로그램 흐름의 통제권 및 객체 생명주기 관리 권한이 개발자로부터 외부(프레임워크, 컨테이너)로 넘어가는 현상을 제어의 역전(IoC)이라고 함

#### 2) 스프링 프레임워크에서의 IoC
- 개발자가 XML 또는 자바 설정 클래스에 사용할 객체에 대한 정보를 기재해 두면, 스프링 컨테이너가 해당 정보를 바탕으로 컴포넌트(객체)들을 직접 생성하고 의존성을 주입하여 프로그램 흐름을 제어함

## 4.2 의존성 주입 (DI)
- 의존성 주입(DI, Dependency Injection)
    - 객체가 스스로 자기에게 필요한 협력 객체(의존성)를 new 연산자로 선언해 만들지 않고, 외부에서 생성된 객체를 전달받아 사용하는 객체지향 설계 기법
    - 의존성을 제공하는 주체(스프링 컨테이너)와 의존성을 사용하는 객체 간의 결합도를 낮추는 효과를 제공

### 4.2.1 설정 클래스(AppConfig) 구현
빈 생성을 전담 제어할 수 있는 설정 클래스를 정의
```java
@Configuration // 스프링 컨테이너 설정 정보 클래스 정의
public class AppConfig {
  
  @Bean // 스프링 빈 등록 명시 (메서드명인 car가 빈의 고유 이름이 됨)
  public Car car() {
    return new GasolineCar(); // 필요 시 new HybridCar() 로 손쉽게 변경 가능
  }

  @Bean // driver 빈 등록 및 의존성 수동 전달
  public Driver driver(Car car) { // 1차로 Car 타입의 빈을 찾고, 매개변수명 car라는 이름의 빈을 찾음
    return new Driver(car); // 위에서 등록된 car 빈을 직접 주입받아 객체 생성
  }
}
```

### 4.2.2 스프링 컨테이너 구동 및 빈 조회
메인 실행 클래스에서 프레임워크가 제공하는 `어노테이션 설정 클래스 기반 컨테이너`(AnnotationConfigApplicationContext)를 사용하여 설정 정보를 로딩하고 조립된 빈을 검색해 실행하는 흐름 구성.

```java
public class SpringCoreApplication {
  public static void main(String[] args) {
    // 1. 순수 스프링 프레임워크의 자바 설정 컨테이너를 수동 구동 (AppConfig 설정 정보 적재)
    AnnotationConfigApplicationContext context = new AnnotationConfigApplicationContext(AppConfig.class);

    // 2. 스프링 컨테이너에서 최종 완성 조립된 Driver 빈 조회
    Driver driver = context.getBean(Driver.class);

    // 3. 비즈니스 기능 가동
    driver.driveCar();

    // 4. 컨테이너 종료 및 리소스 반환
    context.close();
  }
}
```

## 4.3 관점 지향 프로그래밍 (AOP) 기초

### 4.3.1 AOP(Aspect Oriented Programming) 개념과 도입 배경
- OOP의 한계 극복:
    - 객체지향 프로그래밍(OOP)은 비즈니스 요구사항을 모듈화하여 관리하기에 우수하지만, 여러 클래스에 걸쳐 공통적으로 나타나는 부가 기능(로깅, 보안, 트랜잭션 등)의 중복 코드를 깔끔하게 분리하기 어려운 한계가 존재함
    - 이처럼 애플리케이션의 핵심 비즈니스 로직 이외에 여러 영역에 흩어져 중복 수행되는 공통 부가 기능들을 횡단 관심사(Cross-cutting Concern)라고 정의함
    - 핵심 비즈니스 로직(Core Concern)과 이 횡단 관심사를 분리하여 독립적인 모듈로 정비해 관리하는 설계 기법을 관점 지향 프로그래밍(AOP)이라고 함
- AOP 핵심 용어:
    - Aspect: 분리해 낸 공통 관심사를 하나로 묶어 독립적으로 설계한 모듈 단위
    - Target: 부가 기능(Aspect)을 부여받는 실질적인 비즈니스 핵심 클래스
    - JoinPoint: 부가 기능이 삽입되어 동작할 수 있는 프로그램 실행 과정 상의 지점이나 타이밍 (메서드 호출, 객체 생성 시점 등)
    - Pointcut: 수많은 JoinPoint 중에서 실제로 Aspect 부가 기능을 적용할 대상 지점들을 지정하는 필터링 패턴 규칙
    - Advice: Pointcut으로 지정된 대상 지점에서 실제로 수행할 구체적인 부가 기능 구현 코드 블록

### 4.3.2 Advice의 5가지 동작 시점
스프링 AOP는 타깃 메서드의 실행 주기에 맞춰 부가 기능이 끼어들 위치를 결정하는 5가지 종류의 Advice를 지원함
- @Before (이전): 타깃 메서드가 실행되기 직전에 부가 기능을 가동하는 형태
- @AfterReturning (정상 반환 이후): 타깃 메서드가 예외 발생 없이 성공적으로 실행을 끝마치고 결과 값을 반환한 직후에 가동하는 형태
- @AfterThrowing (예외 발생 이후): 타깃 메서드 실행 도중 예외가 발생하여 예외가 던져지는 시점에 가동하는 형태
- @After (이후): 타깃 메서드의 성공 여부와 무관하게 메서드가 끝나면 무조건 실행되는 사후 처리 형태 (try-catch의 finally 블록과 유사함)
- @Around (전후 주변): 타깃 메서드의 호출 전후 과정을 완전히 장악하여 실행 시점 통제, 반환값 조작, 예외 가로채기 등 가장 강력한 제어를 지원하는 형태

### 4.3.3 프록시 기반 AOP 기술
- 프록시 (Proxy) 개념:
    - 클라이언트가 실제 비즈니스 객체(Target)를 직접 호출하는 것이 아니라, 스프링이 중간에 가짜 대리인 객체(Proxy)를 생성하여 가로채기 구조를 만듦
    - 대리인 객체가 호출을 받아 공통 업무(Advice)를 처리하고 진짜 객체를 호출해 주는 대행 메커니즘을 기반으로 AOP를 달성함
- 스프링이 사용하는 두 가지 프록시 생성 기술:
    - JDK Dynamic Proxy: 자바 표준 API를 사용하여 프록시를 동적 생성하는 기법. 반드시 타깃 객체가 인터페이스를 구현하고 있어야 적용 가능함
    - CGLIB(Code Generation Library, 시지엘리브) Proxy: 외부 바이트코드 조작 라이브러리를 사용하여 타깃 구체 클래스를 상속받는 프록시 객체를 하부에서 동적으로 복제 생성하는 기법. 인터페이스 유무와 무관하게 작동하므로, 현대 스프링 프레임워크와 스프링 부트의 기본 프록시 생성 기술로 공식 채택되어 사용됨

```java
// 실제 자동차와 동일한 인터페이스를 구현하여 대리인(Proxy) 역할을 수행
public class ProxyCar implements Car {
    private final Car target; // 진짜 자동차 객체를 품고 있음 (Target)

    public ProxyCar(Car target) {
        this.target = target;
    }

    @Override
    public void startEngine() {
        // Before Advice 동작 모방 (공통 로깅)
        System.out.println("[AOP 로그] 자동차 시동을 걸기 전에 엔진 점검을 시작합니다.");
        
        target.startEngine(); // 진짜 객체의 기능 수행
    }

    @Override
    public void drive() {
        long start = System.currentTimeMillis(); // Around Advice 이전 처리 (시간 측정)
        
        target.drive(); // 진짜 객체의 기능 수행
        
        long end = System.currentTimeMillis(); // Around Advice 이후 처리 (시간 계산)
        System.out.println("[AOP 로그] 주행 완료 소요 시간: " + (end - start) + "ms");
    }

    @Override
    public void stopEngine() {
        target.stopEngine();
        
        // After Advice 동작 모방
        System.out.println("[AOP 로그] 주행이 안전하게 종료되었습니다.");
    }
}
```

- 조립 및 가동 테스트 (AppConfig에서 진짜 객체 대신 프록시 대리인을 반환하도록 수동 조립):
```java
@Configuration
public class AppConfig {
    @Bean
    public Car car() {
        Car realCar = new ElectricCar(); // 진짜 핵심 객체 생성
        return new ProxyCar(realCar); // 스프링 빈으로 가짜 대리인(Proxy)을 대신 등록하여 우회 주입되도록 통제
    }

    @Bean
    public Driver driver() {
        return new Driver(car()); // ProxyCar가 주입되어 동작함
    }
}
```

# 5. 스프링 컨테이너 핵심 메커니즘

## 5.1 스프링 빈 스코프 (Scope)
- 스코프(Scope)는 스프링 빈 객체가 스프링 컨테이너 내부에서 유지되고 유효성을 가지는 생존 범위.
- 싱글톤 스코프 (Singleton Scope): 컨테이너 구동 단계에서 빈을 단 하나만 사전 생성하여 전체 사용처에서 하나의 객체를 공유하는 스프링의 기본 스코프 방식. 상태를 저장하지 않는 무상태 방식으로 빈을 설계해야 멀티 스레드 가동 환경에서 안전한 동작을 보증할 수 있음.
- 프로토타입 스코프 (Prototype Scope): 빈 조회 요청이 접수될 때마다 컨테이너가 새로운 빈을 독립적으로 매번 생성하여 반환하는 특수 스코프 방식. 스프링 컨테이너는 프로토타입 빈의 생성 및 의존 관계 주입, 초기화 콜백까지만을 책임지며, 소멸 처리는 빈을 반환받은 클라이언트 코드가 주도적으로 관장해야 함.

## 5.2 스프링 빈 생명주기 및 콜백
### 5.2.1 생명주기 콜백의 필요성
- 애플리케이션 시작(구동) 시점에 데이터베이스 커넥션 풀을 사전 연결해 두거나 외부 네트워크 소켓 세션을 연결하는 작업, 그리고 서버가 정지(종료)되기 직전에 열려 있던 커넥션을 부드럽게 닫고(Graceful Shutdown) 메모리 버퍼 자원을 안전하게 반환하는 등 외부 인프라스트럭처 리소스를 연동 제어할 때 사용함
- 단순 자바 객체는 인스턴스가 생성되는 시점(생성자 실행)에는 아직 스프링 컨테이너의 의존성 주입이 완료되지 않아 협력 객체를 참조할 수 없으므로, 주입이 끝난 직후를 안전하게 보증받는 초기화 시점과 소멸 시점을 잡기 위해 콜백 기술을 적용함

### 5.2.2 스프링 빈 생명주기 전체적인 흐름
- 1. 스프링 컨테이너 기동
- 2. 스프링 빈 인스턴스 생성
- 3. 의존 관계 주입 완료
- 4. 초기화 콜백 메서드 수행
- 5. 빈의 기동 및 비즈니스 로직 수행
- 6. 소멸 콜백 메서드 수행
- 7. 스프링 컨테이너 종료 및 리소스 소멸

### 5.2.3 빈의 생명주기 콜백 지정 방식
- 어노테이션 지정 방식: 빈 내부 실행 메서드 상단에 PostConstruct(기동 초기화) 및 PreDestroy(소멸 전 정리) 표준 어노테이션 기입 및 지정. (현대 자바 진영 및 스프링 진영의 최우선 표준 권장안)
- 인터페이스 상속 방식: InitializingBean(afterPropertiesSet 구현) 및 DisposableBean(destroy 구현) 인터페이스 직접 구현을 통한 제어. (프레임워크 기술 규격에 코드가 강력 종속되므로 현대적인 구조에서는 지양됨)
- 빈 설정 옵션 방식: 자바 빈 설정 작성 시 Bean(initMethod = "메서드명", destroyMethod = "메서드명") 형태로 콜백 이름 기입 및 지정.

```java
import jakarta.annotation.PostConstruct;
import jakarta.annotation.PreDestroy;

public class TempFileSupport {
    private String tempFilePath;

    public TempFileSupport() {
        System.out.println("생성자 호출: 객체가 생성됨. 임시파일경로 = " + tempFilePath);
    }

    public void setTempFilePath(String tempFilePath) {
        this.tempFilePath = tempFilePath;
    }

    // 1. 초기화 콜백 (의존 관계 주입이 끝난 직후 가동되어 임시 작업 파일 생성)
    @PostConstruct
    public void initTempFile() {
        System.out.println("초기화 콜백: 작업용 임시 파일 생성 완료. 경로 = " + tempFilePath);
    }

    // 2. 비즈니스 기능 실행
    public void writeLog(String logMessage) {
        System.out.println("임시 파일에 로그 기록: " + logMessage);
    }

    // 3. 소멸 콜백 (컨테이너 종료 직전에 가동되어 자원 해제 및 파일 자동 삭제)
    @PreDestroy
    public void clearTempFile() {
        System.out.println("소멸 콜백: 임시 파일 안전하게 삭제 및 디스크 자원 반환.");
    }
}
```

- 설정 클래스 등록 (AppConfig):
```java
@Configuration
public class AppConfig {
    @Bean
    public TempFileSupport tempFileSupport() {
        TempFileSupport support = new TempFileSupport();
        support.setTempFilePath("resources/temp.log"); // 가상의 임시 파일 경로 주입
        return support;
    }
}
```

- 메인 구동 및 로그 검증 (SpringCoreApplication):
```java
public class SpringCoreApplication {
    public static void main(String[] args) {
        // 1. 컨테이너 기동 및 설정 정보 로딩 (초기화 콜백 자동 발동)
        AnnotationConfigApplicationContext context = 
                new AnnotationConfigApplicationContext(AppConfig.class);

        // 2. 빈 조회 및 비즈니스 사용
        TempFileSupport support = context.getBean(TempFileSupport.class);
        support.writeLog("사용자 로그인 작업 수행 로그 기록.");

        // 3. 컨테이너 종료 (소멸 콜백 자동 발동)
        context.close();
    }
}
```

```text

```
