# 스프링 프레임워크 핵심 원리

## 학습 목표
- 프레임워크와 라이브러리의 개념 및 제어의 역전(IoC) 원리를 구분함.
- 빌드 도구인 Gradle과 Maven의 구조를 분석함.
- 스프링 프레임워크 핵심 설계 원칙인 제어의 역전(IoC), 의존성 주입(DI), 관점 지향 프로그래밍(AOP)의 탄생 배경과 구동 원리를 이해함.
- 스프링 빈의 생성과 소멸에 이르는 전체 생명주기와 이를 제어하는 스프링 컨테이너의 관리 절차를 파악함.
- 프록시 디자인 패턴의 개념과 스프링 AOP의 대리인 객체 구동 방식을 파악함.
- 기술 교체 시에도 비즈니스 코드를 보존하는 서비스 추상화(PSA)의 개념적 유연성을 이해함.

# 1. 스프링 입문

## 1.1 프레임워크와 라이브러리

### 1.1.1 라이브러리(Library)
- 애플리케이션 개발에 필요한 특정 기능들이 미리 구현된 재사용 가능한 코드, 클래스, 함수의 집합.
- 날짜 계산이나 데이터 암호화처럼 복잡하고 반복적인 기능 코드를 매번 직접 설계하지 않고, 이미 품질과 신뢰성이 보장된 리소스를 활용하여 전체 결함을 낮추고 개발 생산성을 높이기 위해 필요함.
- 개발자가 전체적인 프로그램의 실행 흐름을 직접 제어하며, 필요한 시점에 적절한 라이브러리를 주도적으로 호출하여 사용함.
- 칼과 도마, 냄비만 사용하여 요리하던 환경(자바의 표준 API만 사용)에서 필요에 따라 믹서기, 오븐, 전자레인지처럼 요리를 단축해 주는 조리 도구(라이브러리)를 이용해서 직접 조리하는 환경.
- 냉동실에서 고기를 꺼내서 자연 해동을 하든, 전자레인지의 해동 버튼을 누르든, 요리의 주체인 개발자가 원하는 도구를 직접 선택하고 활용하는 것처럼, 코드의 실행 흐름과 제어 권한이 개발자에게 있음.

### 1.1.2 프레임워크(Framework)
- 애플리케이션을 완성하는 데 필요한 전체적인 설계 구조와 구동 규칙을 기본 뼈대로 미리 만들어 제공하고, 개발자가 세부적인 기능 코드를 약속된 위치에 끼워 넣으면 프로그램이 가동될 때 적절한 타이밍에 개발자의 코드를 호출하여 전체 실행 흐름을 통제해 주는 소프트웨어 개발의 기반이 되는 플랫폼.
- 다수의 작업자가 참여하는 거대 프로젝트에서 프로그램의 기본 뼈대와 규칙을 강제함으로써, 개발자 개개인의 설계 차이에서 오는 품질 격차를 최소화하고 코드를 일관되게 정립하기 위해 필요함.
- 데이터 접근이나 사용자 인증 등 매번 공통 설계가 필요한 작업은 프레임워크에 위임하여 생산 효율을 높이고 개발자는 핵심 비즈니스 구현에만 집중할 수 있음.
- 개발자는 프레임워크가 정의한 설계 규칙에 따라 코드를 작성하여 삽입하며, 프레임워크가 이 코드를 적절한 타이밍에 호출하여 실행함.
- 전문 요리사를 고용한 사람이 식재료를 약속된 위치에 준비해 두기만 하면, 요리사는 본인이 사용할 전용 조리도구까지 스스로 챙겨서 요리 중 필요한 시점에 재료를 가져가서 조리하는 흐름과 유사함.
- 전체 요리 진행 순서와 조리도구 세팅 및 관리 등 작동 흐름은 요리사가 관장하며, 개발자는 지정된 장소에 원하는 식재료를 채워두는 역할에만 집중함.
  - 제어의 역전 (Inversion of Control)
    - 개발자가 직접 객체를 생성하고 코드의 실행 흐름을 주도하는 것이 아니라, 프레임워크가 객체의 생명주기를 관리하고 작성된 코드를 알맞은 타이밍에 실행하여 흐름을 통제하는 소프트웨어 설계 원칙.
    - 스프링 프레임워크에서는 스프링 컨테이너가 객체의 생성 및 소멸, 관계 설정 처리를 대행하여, 개발자가 직접 흐름을 통제하지 않고 핵심 비즈니스 로직[^1] 설계에만 전념할 수 있도록 지원함.
    - 영화배우 오디션에서 담당자들이 지원자에게 자주 언급하는 헐리우드 원칙("Don't call us, we'll call you.")에서 유래한 소프트웨어 설계 규칙.
    - 개발자가 프레임워크의 코드를 직접 호출하여 가동하지 않고, 정해진 규칙에 따라 코드를 등록해 두면 프레임워크가 필요할 때 해당 코드를 호출함.

## 1.2 스프링 프레임워크 개요
- 자바 엔터프라이즈 애플리케이션 개발을 편리하게 해주는 오픈소스 프레임워크.
- 주요 장점:
  - 데이터베이스 연결이나 사용자 인증처럼 번거롭고 복잡한 기본 설정 및 인프라 구축 작업을 대신 해결해 주어, 개발자가 핵심 비즈니스 로직[^1]<sup>2</sup> 개발에만 집중하도록 지원.

  - 필요한 기능들이 부품처럼 미리 규격화되어 있어서, 약속된 규칙에 맞춰 조립하듯이 빠르게 대규모 웹 서비스 시스템 신속 구축.
  - 데이터베이스 종류나 연동하는 외부 시스템의 기술 규격이 도중에 바뀌더라도, 이미 작성해 둔 프로그램 코드를 고치지 않고 부품만 유연하게 교체.
- 주요 단점:
  - 제공하는 기술 범위가 광범위하고 핵심 개념이 추상적이어서, 초보 개발자가 구동 원리를 완벽히 파악하기 전까지 학습 장벽이 존재함.
  - 프로젝트를 시작할 때 수많은 구성 파일이나 라이브러리 설정을 수동으로 정밀하게 제어해야 하므로, 초기 환경을 구축하는 절차가 매우 복잡함.
- 과거 자바 진영의 기술 규격이었던 EJB(Enterprise JavaBeans)는 보안과 분산 처리 기능을 제공했으나, 간단한 기능 하나를 만들 때도 수많은 설정 파일 작성과 무거운 인터페이스 상속을 강제하여 개발 생산성이 저하되는 원인으로 지목.
- 스프링은 기술 종속성 없이 가볍고 순수한 자바 기본 객체인 POJO(Plain Old Java Object)만을 사용하여 유연한 객체지향 설계 지원.

## 1.3 스프링 핵심 용어 사전
- 의존성 (Dependency)
  - 한 객체가 자신의 역할을 수행하는 과정에서 다른 객체를 호출하거나 참조하여 작동을 의지하는 협력 관계
  - 예를 들어 운전자(Driver) 객체가 정상 동작하기 위해 자동차(Car) 객체를 필요로 하는 것처럼, 하나의 컴포넌트가 작동하기 위해 필수적으로 결합되어야 하는 대상 객체나 리소스
- 의존성 주입 (DI, Dependency Injection)
  - 어떤 객체가 필요로 하는 협력 객체를 스스로 직접 생성하여 확보하지 않고, 외부의 관리 장치(컨테이너)로부터 주입(전달)받아 사용하는 설계 기법
  - 객체 간 결합도를 낮추고 모듈 독립성을 높여 기술의 사후 변경을 용이하게 유도
- 스프링 빈 (Spring Bean)
  - 스프링 컨테이너가 스스로 생성, 관계 조립, 생명주기 관리를 전담하는 대상 자바 객체
  - 개발자가 직접 new 연산자로 선언하여 사용하는 일반 자바 객체와 구별되며, 컨테이너에 의해 제어권이 위임된 컴포넌트만을 지칭
- 스프링 컨테이너 (Spring Container / IoC Container)
  - 스프링 빈의 인스턴스화, 구성설정, 생명주기 제어 및 의존 관계 연결 처리를 담당하는 핵심 구동 엔진
  - 개발자가 작성한 컴포넌트들을 유기적으로 조립하는 자동화 공장 역할을 수행
- 설정 메타데이터 (Configuration Metadata)
  - 스프링 컨테이너가 어떤 객체를 생성하고 어떻게 의존 관계를 이어 붙여 조립할지 지시하는 조립 설계도 문서
  - 과거에는 XML 문서 형태로 기재했으나, 현대 스프링 표준에서는 자바 설정 클래스 파일 상단에 @Configuration, @Bean 등의 어노테이션 지시어를 부여하여 관리

# 2. 빌드 도구 (Build Tool)

## 2.1 스프링 프로젝트의 빌드 및 배포 흐름

### 2.1.1 빌드와 배포
- 빌드(Build)의 정의
  + 개발자가 작성한 자바 소스코드를 컴퓨터가 실행할 수 있는 최종 소프트웨어 산출물(.jar)로 변환하는 일련의 작업 과정
- 빌드의 세부 과정
  + 1. 의존성 해결(Dependency Resolution): 빌드 설정 파일(build.gradle 등)을 파싱[^2]하여 프로젝트 가동에 필요한 외부 라이브러리들을 원격 저장소에서 다운로드
  + 2. 컴파일(Compile): 사람이 읽기 쉬운 자바 소스코드(.java)를 자바 가상 머신(JVM)이 실행할 수 있는 바이트코드(.class)로 번역함
  + 3. 테스트 실행(Test): JUnit 등의 테스트 프레임워크를 가동하여 작성된 비즈니스 로직의 결함 유무를 빌드 완료 전에 최종 검증함
  + 4. 패키징(Packaging): 컴파일을 마친 바이트코드와 리소스 파일(properties, yml 등)을 단일 실행 가능한 형태(.jar)로 압축함
- 배포(Deploy)의 정의
  + 빌드를 통해 생성된 최종 소프트웨어 산출물(.jar)을 실제 서비스를 제공할 운영 서버 환경에 올리고 실행하여 사용자가 서비스를 이용할 수 있도록 만드는 과정

### 2.1.2 스프링 프로젝트의 빌드와 배포 절차
- 소스코드 작성: 개발자가 로컬 PC에서 요구사항에 맞는 자바 프로그램 소스코드와 애플리케이션 설정 파일을 작성함
- 빌드 도구 가동: Gradle 또는 Maven 빌드 도구를 실행하여 작성된 소스코드를 컴파일하고 오류 검증용 테스트를 통과시킨 뒤, 최종 실행 파일(.jar)로 압축 패키징함
- 실행 파일(jar) 전송: 로컬 PC 빌드로 산출된 jar 실행 파일을 실제 서비스를 운영할 원격 서버나 클라우드 서버 환경으로 전송함
- 배포 및 service 실행: 전송받은 서버 환경에서 java -jar 명령어로 애플리케이션을 단독 구동하여 사용자의 실제 요청을 처리할 수 있도록 서비스를 개시함

## 2.2 Maven과 Gradle
- 외부 라이브러리를 다운로드하고 소스코드를 컴파일하여 최종 실행 파일로 패키징하는 빌드 자동화 도구.

### 2.2.1 Maven (메이븐)
#### 1) 개요
- 아파치 재단[^3]에서 개발한 자바 프로젝트 빌드 도구로, XML[^4] (pom.xml) 기반의 정형화된 라이프사이클 모델을 따름.

#### 2) 특징
- 프로젝트 설정과 의존성[^5] 정보를 XML 형태로 엄격하게 표현하며, 설정 방식이 규격화되어 일관된 구조로 관리할 수 있음.

#### 3) 한계
- 프로젝트 규모가 커질수록 XML 구조가 비대해지고 태그 중복이 증가하여 가독성이 저하되며, 빌드 스크립트[^6] 내 동적 논리 제어가 어려움.

### 2.2.2 Gradle (그레이들)
#### 1) 개요
- Groovy 또는 Kotlin DSL(Domain Specific Language) 설정 언어를 탑재하여, 메이븐의 선언적 설정[^7] 장점과 프로그래밍 유연성을 결합한 표준 빌드 도구.

#### 2) 특징
- 단순 설정을 넘어 빌드 스크립트[^6] 내에서 변수 선언 및 제어문 활용 등 동적인 빌드 스크립트 작성 지원.

#### 3) 성능적 장점
- 이전 빌드 결과물을 재사용하고, 변경이 발생한 소스 파일만 선별적으로 컴파일하는 증분 빌드 기법을 적용하여 빌드 속도를 개선.

### 2.2.3 의존성 관리 (Dependency Management)
- 설정 파일에 라이브러리 명칭만 명시하면 빌드 도구가 네트워크를 통해 필요한 라이브러리 자동 다운로드.

- repositories: 프로젝트에 필요한 외부 라이브러리들을 받아올 서버 저장소를 지정하며, mavenCentral()은 메이븐 중앙 라이브러리 저장소를 지칭
- dependencies: 프로젝트 컴파일 및 테스트 구동에 사용할 외부 라이브러리들을 그레이들이 다운로드하도록 선언하는 영역으로, 기본적으로 단위 테스트를 위한 junit-jupiter 라이브러리가 명시되어 있음
- test: 테스트 실행 환경을 지정하는 태스크 설정 영역으로, useJUnitPlatform()을 설정하여 JUnit 5 테스트 엔진으로 테스트가 실행되도록 지정

### 3.4.2 의존성 로드 및 빌드 갱신
의존성을 반영하여 외부 라이브러리를 프로젝트에 적용하려면 그레이들 동기화(Sync) 작업을 수행해야 함.
- 방법 1 (편집창 단추 이용): build.gradle 파일을 수정한 뒤 자바 편집창 우측 상단 영역에 일시적으로 노출되는 파란색 새로고침 기호가 표시된 작은 코끼리 모양의 Gradle 변경 사항 로드(Sync Gradle Changes) 단추를 클릭
- 방법 2 (Gradle 탭 이용): 인텔리제이 화면 우측 가장자리 세로 툴바에서 코끼리 아이콘(Gradle)을 클릭하여 Gradle 탭 창을 활성화한 뒤, 탭 창 내부의 도구 모음 맨 좌측에 위치한 회전 화살표(새로고침) 모양의 Sync All Gradle Projects 단추를 클릭
- 다운로드 확인: 그레이들이 메이븐 중앙 저장소로부터 spring-context 라이브러리와 이에 연동되는 스프링 코어 모듈들을 자동으로 다운로드하여 컴파일 클래스 패스에 정상 적재하는지 최종 확인
  - Project 창의 SpringCore 프로젝트 아래에 External Libraries를 확장해서 `Gradle: org.springframework:spring-xxx:7.0.8` 라이브러리 들이 추가 되었는지 확인

## 4.2 다형성을 활용한 느슨한 결합
```java
interface Car {
    void startEngine();
    void drive();
    void stopEngine();
}

class GasolineCar implements Car {
    @Override
    public void startEngine() {
        System.out.println("가솔린 엔진에 점화하여 시동을 켭니다.");
    }
    @Override
    public void drive() {
        System.out.println("가솔린 동력으로 주행합니다.");
    }
    @Override
    public void stopEngine() {
        System.out.println("가솔린 공급을 차단하여 시동을 끕니다.");
    }
}

class HybridCar implements Car {
    @Override
    public void startEngine() {
        System.out.println("시스템 전원을 인가하고 하이브리드 시동을 켭니다.");
    }
    @Override
    public void drive() {
        System.out.println("전기와 엔진을 혼합하여 효율적으로 주행합니다.");
    }
    @Override
    public void stopEngine() {
        System.out.println("하이브리드 시스템 종료 처리를 작동합니다.");
    }
}

class Driver {
    private Car car; // 인터페이스에 의존 (느슨한 결합)

    // 생성자를 통해 외부에서 Car 구현체를 주입 받음
    public Driver(Car car) {
        this.car = car;
    }

    public void driveCar() {
        car.startEngine();
        car.drive();
        car.stopEngine();
    }
}
```
- 새로운 클래스(예: ElectricCar)를 정의하여 구현하고, 외부에서 주입해 주는 객체만 바꿔 끼워주면 운전자 코드는 그대로 재사용함

- 💻 실습 (OOP를 적용하기 이전 Driver, GasolineCar): [oop/before](../SpringCore/src/main/java/net/likelion/bebc25/oop/before)
- 💻 실습 (OOP를 적용한 후 Driver, GasolineCar): [oop/after](../SpringCore/src/main/java/net/likelion/bebc25/oop/after)

## 4.3 제어의 역전(IoC)의 출현과 개념
### 4.3.1 제어 주도권의 양도
- [직접 결합 방식](#41-의존-객체를-직접-결합하는-방식)에서는 개발자가 Driver 클래스 내부에서 직접 new GasolineCar() 또는 new HybridCar()를 기재하여 협력 객체의 생성과 의존성 조립을 스스로 통제함
- [느슨한 결합 방식](#42-다형성을-활용한-느슨한-결합)을 적용하면 Driver 클래스는 단지 Car 인터페이스에만 의존하며, 구체적인 구현 객체의 생성과 주입 권한은 외부로 넘어감
- 이처럼 프로그램 흐름의 통제권 및 객체 생명주기 관리 권한이 개발자로부터 외부(프레임워크, 컨테이너)로 넘어가는 현상을 제어의 역전(IoC)이라고 함

### 4.3.2 스프링 프레임워크에서의 IoC
- 개발자가 XML 또는 자바 설정 클래스에 사용할 객체에 대한 정보를 기재해 두면, 스프링 컨테이너가 해당 정보를 바탕으로 컴포넌트(객체)들을 직접 생성하고 의존성을 주입하여 프로그램 흐름을 제어함

# 5. 의존성 주입 (DI: Dependency Injection)
- 객체가 스스로 자기에게 필요한 협력 객체(의존성)를 new 연산자로 선언해 만들지 않고, 외부에서 생성된 객체를 전달받아 사용하는 객체지향 설계 기법
- 의존성을 제공하는 주체(스프링 컨테이너)와 의존성을 사용하는 객체 간의 결합도를 낮추는 효과를 제공

## 5.1 설정 클래스 정의
빈 생성을 전담 제어할 수 있는 설정 클래스를 정의

```java
@Configuration // 스프링 컨테이너 설정 정보 클래스 정의
public class AppConfig {

    @Bean // 스프링 빈 등록 명시 (메서드명인 car가 빈의 고유 이름이 됨)
    public Car car() {
        return new GasolineCar(); // 필요 시 new HybridCar() 로 손쉽게 변경 가능
    }

    @Bean // driver 빈 등록 및 의존성 수동 전달
    public Driver driver(Car car) { // Car 타입의 빈을 찾아서 자동으로 주입
        return new Driver(car); // 위에서 등록된 car 빈을 직접 주입받아 객체 생성
    }
}
```

## 5.2 메인 클래스 구성
- 스프링 프레임워크가 제공하는 AnnotationConfigApplicationContext를 사용하여 설정 정보를 로딩하고 조립된 빈을 검색해서 실행

```java
public class SpringCoreApplication {
    public static void main(String[] args) {
        // 1. 스프링 컨테이너 생성 (AppConfig 클래스 전달)
        AnnotationContext context = new AnnotationConfigApplicationContext(AppConfig.class);

        // 2. 스프링 컨테이너에서 조립된 Driver 빈 조회
        Driver driver = context.getBean(Driver.class);

        // 3. 비즈니스 기능 호출
        driver.driveCar();
    }
}
```

## 5.3 의존성 주입 방식
### 5.3.1 생성자 주입과 필드 주입 방식
의존성을 주입받는 통로(접점)에 따라 생성자 주입, 수정자 주입, 필드 주입 기법이 사용됨.

#### 1) 생성자 주입 (Constructor Injection)
- 객체 생성 시점에 생성자를 통해 필요한 의존 관계를 주입받는 방식
- 장점
  - 객체 생성 시 딱 한 번만 호출되므로 런타임에 의존 객체가 바뀔 위험이 없어 불변성(Immutability) 확보 가능
  - 자바 final 키워드를 선언할 수 있어 의존성 주입 누락 시 컴파일러 단계에서 에러로 검증 가능
  - 순수 자바 코드로 단위 테스트 작성 시 생성자에 mock 객체를 주입하기 용이함
- 단점
  - 의존 관계가 가변적이거나 선택적인 경우(런타임에 유연하게 변해야 하는 경우) 대응하기 어려움

  - 💻 실습 (스프링 DI를 이용한 의존성 주입 - Constructor Injection): [spring/di/constructor](SpringCore/src/main/java/net/likelion/bebc25/spring/di/constructor)

#### 2) 수정자 주입 (Setter Injection)
- setter 메서드를 정의하여 필요한 의존 관계를 주입받는 방식
- 장점
  - 의존 객체가 필수적이지 않거나 런타임에 유연하게 변경이 필요한 가변 관계에 적합함
- 단점
  - setter 메서드가 public으로 노출되므로 의존 객체가 임의로 변경될 위험이 존재함
  - 주입을 누락해도 객체가 정상적으로 생성되어, 이후 비즈니스 호출 시 NullPointerException 발생 위험이 높음

  - 💻 실습 (스프링 DI를 이용한 의존성 주입 - Setter Injection): [spring/di/setter](SpringCore/src/main/java/net/likelion/bebc25/spring/di/setter)

#### 3) 필드 주입 (Field Injection)
- 멤버 변수 상단에 의존 주입 어노테이션을 부착하여 스프링 컨테이너가 직접 값을 할당하도록 유도하는 방식
- 장점
  - 별도의 생성자나 setter 작성 없이 어노테이션 한 줄로 의존 관계를 선언하므로 코드가 간결함
- 단점
  - 스프링 컨테이너가 구동되지 않는 순수 자바 단위 테스트 환경에서 의존 객체를 수동 조립할 방법이 없어서 테스트 작성이 어려움
  - 프레임워크 기술 규격에 과도하게 밀착 종속되므로 컴포넌트 독립성이 훼손되는 안티 패턴에 해당함

### 5.3.2 생성자 주입을 권장하는 이유
현대 스프링 개발 구조에서는 생성자 주입만을 표준 설계 기법으로 공식 권장함.
- 객체의 불변성(Immutability) 보장: 의존 관계가 조립된 이후 런타임에 외부 공격이나 실수로 인해 주입 객체가 변경되는 사고를 컴파일 수준에서 원천 차단함
- 컴파일 타임 의존성 검증: final 필드를 누락한 채 생성자를 구현하면 컴파일러가 오류를 잡아주므로 널 포인터 예외(NPE)를 완전 예방함
- 순수 자바 테스트 용이성: 프레임워크(스프링 컨테이너) 구동 없이도 테스트 코드에서 수동으로 new Driver(mockCar) 처럼 순수 자바 객체를 조립해서 테스트가 가능함

### 5.3.3 순환 참조 문제와 방지 대책
두 개 이상의 컴포넌트가 서로를 멤버 필드로 품어 상호 의존하는 상태를 순환 참조라고 함.
- 순환 참조 발생 시 현상
  - 필드 주입이나 수정자 주입을 사용하면, 스프링 컨테이너 가동 및 객체 생성과 의존성 주입 단계에서는 문제가 없지만 비즈니스 메서드가 호출되어 서로를 순환 참조할 때 비로소 스택오버플로우(StackOverflowError)를 발생시켜 런타임 장애를 초래함
- 생성자 주입의 해결책
  - 생성자 주입을 채택하면 스프링 컨테이너 구동(인스턴스 생성 및 빈 등록) 단계에서 생성자가 상대 객체를 서로 필요로 하는 상황인 순환 참조 관계에 놓인 객체들을 감지하고 즉시 구동을 거부하여 `BeanCurrentlyInCreationException` 예외를 발생시키므로 개발 환경에서 미리 예방 및 제거 조치가 가능함

# 6. 관점 지향 프로그래밍 (AOP: Aspect Oriented Programming)

## 6.1 AOP 개념과 도입 배경
#### 1) 핵심 관심사와 횡단 관심사
- 애플리케이션의 핵심 가치와 비즈니스 목적을 나타내는 비즈니스 관심 영역을 핵심 관심사(Core Concern)라고 함
- 핵심 관심사 이외에 여러 모듈과 클래스 전반에 걸쳐 공통적으로 흩어져 반복 수행되는 부가 기능(로깅, 보안, 성능 측정, 트랜잭션, 에러 처리 등)을 횡단 관심사(Cross-cutting Concern)라고 함

#### 2) OOP만 사용할 경우의 문제점
- 코드 중복성 증가: 동일한 횡단 관심사 코드가 여러 클래스와 메서드에 반복 분산 선언되면서 전체 프로젝트의 중복 코드 비중을 증가시킴
- 코드 가독성 저하: 핵심 비즈니스 로직 사이에 공통 부가 기능 코드가 혼재되면서 가독성을 가로막고, 이로 인해 잠재적 결함 및 오류 식별에 어려움을 초래함
- 개발 생산성 감소: 비즈니스 도메인 분석에 주력해야 할 애플리케이션 개발자가 인프라 제어 목적의 공통 부가 코드를 매번 직접 작성해야 하므로 핵심 개발 업무의 집중력이 떨어짐
- 코드 재사용성 저하: 공통 부가 로직이 특정 객체 내에 밀접하게 결합하여 작동하므로, 해당 기능만 따로 분리하여 다른 서비스 컴포넌트에 이식하기 어려움
- 유지보수 난이도 증가: 새로운 횡단 관심 사항의 적용이나 기존 부가 로직의 변경이 발생할 때 여러 클래스를 전부 수정해야 하므로 변경 대응력이 하락함

#### 3) AOP 도입을 통한 한계 극복
- 객체지향 프로그래밍(OOP)은 모듈화 설계에 우수하지만, 횡단 관심사와 핵심 관심사 코드를 원천적으로 분리 정돈하기 어려운 구조적 한계가 존재함
- 이 같은 문제를 극복하기 위해 핵심 관심사와 횡단 관심사를 명확하게 분리하여 각각 독립적인 모듈로 구축하고 관리하는 설계 패러다임인 관점 지향 프로그래밍(AOP)을 도입함

## 6.2 프록시(Proxy) 기반 AOP 기술
### 6.2.1 프록시 (Proxy) 개념
- 의존 객체를 주입할 때 실제 비즈니스 객체(Target)를 전달 하는게 아니라, 대리인 객체(Proxy)를 생성하여 전달
- 대리인 객체가 호출을 받아 횡단 관심을 처리하고 진짜 객체를 호출해 주는 대행 메커니즘을 기반으로 AOP를 달성함

- 💻 실습 (정적 프록시를 이용한 AOP): [spring/aop/staticproxy](SpringCore/src/main/java/net/likelion/bebc25/spring/aop/staticproxy)

### 6.2.2 스프링이 사용하는 두 가지 프록시 생성 기술
#### 1) JDK Dynamic Proxy
- 자바 리플렉션 API의 java.lang.reflect.Proxy 클래스를 사용하여 런타임에 프록시를 동적으로 생성하는 기법. 반드시 타깃 객체가 인터페이스를 구현하고 있어야 적용 가능함
- 스프링 프레임워크의 인터페이스 유무에 따른 프록시 생성 방식
  - 인터페이스가 존재하는 경우: JDK Dynamic Proxy 기술을 기본으로 사용하여 프록시 생성
  - 인터페이스가 없는 경우: CGLIB Proxy 기술을 적용하여 타깃 클래스를 상속받는 프록시 생성
  - 설정을 통해 인터페이스가 있더라도 CGLIB 프록시 적용 강제 가능 (proxyTargetClass 옵션 활성화)
- JDK Dynamic Proxy 제약사항
  - 인터페이스 구현 필수: 인터페이스가 없는 구체 클래스는 동적 프록시 생성 불가
  - 타입 캐스팅 제한: 생성된 프록시 객체는 인터페이스 타입으로만 변환할 수 있어 구현 클래스 타입으로 선언된 빈 주입 시 오류 유발

- 💻 실습 (동적 프록시를 이용한 AOP): [spring/aop/dynamicproxy](SpringCore/src/main/java/net/likelion/bebc25/spring/aop/dynamicproxy)

#### 2) CGLIB(Code Generation Library, 시지엘리브) Proxy
- 외부 바이트코드 조작 라이브러리를 사용하여 타겟 클래스를 상속받는 프록시 객체를 하부에서 동적으로 복제 생성하는 기법. 인터페이스 유무와 무관하게 작동하므로, 현대 스프링 프레임워크와 스프링 부트의 기본 프록시 생성 기술로 공식 채택되어 사용됨
- 스프링 부트의 프록시 생성 정책
  - 인터페이스 존재 여부와 무관하게 기본적으로 CGLIB Proxy 방식으로 프록시 생성
  - 인터페이스 타입 대신 구현 클래스 타입으로 빈을 주입받아 사용할 때 발생하는 타입 캐스팅 예외 예방 목적
- CGLIB Proxy 제약사항
  - final 클래스 상속 불가: final 클래스는 상속을 통한 하부 프록시 객체 생성 불가능
  - final 메서드 재정의 불가: final 메서드는 오버라이딩을 진행할 수 없어 공통 관심사 부가 기능 주입 불가

## 6.3 스프링 AOP와 AspectJ의 관계
#### 1) AspectJ
- 자바 진영의 사실상 표준(de facto standard) AOP 프레임워크
- 컴파일 타임이나 클래스 로딩 타임에 실제 자바 클래스 바이트코드를 수정하여 가동하므로 생성자 호출이나 필드 변경 등 모든 영역에 AOP 개입이 가능함

#### 2) 스프링 AOP
- AspectJ의 편리한 어노테이션(@Aspect, @Before 등) 및 포인트컷 문법을 차용하여 구현한 스프링 내부의 경량 AOP 솔루션
- 스프링이 관리하는 객체(빈)의 메서드 호출 통로에 대리인(프록시)을 세워 가로채는 방식으로 동작하므로 간편하지만 메서드 호출 시점에만 AOP를 적용할수 있다는 한계가 존재함

## 6.4 AOP 핵심 용어

### 6.4.1 Aspect (관점)
- 분리해 낸 횡단 관심사를 하나로 묶어 독립적으로 설계한 클래스
- 클래스 선언부에 @Aspect 지정하여 구현함.

### 6.4.2 JoinPoint (결합 지점)
- Aspect가 삽입되어 동작할 수 있는 프로그램 실행 과정 상의 지점이나 타이밍
- 프록시 방식을 사용하는 스프링 AOP는 스프링 빈의 메서드 실행 시점만을 JoinPoint로 지원함.
- 개발자가 코드상에서 임의로 지정하는 것이 아니며, 스프링 빈 객체의 모든 메서드 호출 시점이 곧 JoinPoint가 됨.

### 6.4.3 Pointcut (필터 지점)
- 수많은 JoinPoint 중에서 실제로 Aspect를 적용할 대상 지점들을 지정하는 필터링 패턴 규칙
- 코드상에서 @Pointcut이나 @Advice 어노테이션에 필터링 패턴 규칙을 기입하여 선언

#### 1) Pointcut 필터 패턴 종류
스프링 AOP는 포인트컷 표현식 내부에 다양한 포인트컷 지시자(PCD: Pointcut Designator)를 선언하여 횡단 관심사를 결합할 대상 지점을 정교하게 필터링함.

- execution
  가장 대표적이고 널리 사용되는 메서드 실행 조인포인트 매칭 지시자. 반환 타입, 패키지 경로, 클래스명, 메서드 시그니처, 파라미터 규격을 종합하여 매칭을 수행함.
  - 예시: `execution(public String net.likelion.bebc25.spring.aop.service.UserService.getUserById(Long))`
  - 설명: net.likelion.bebc25.spring.aop.service 패키지 UserService 클래스에 선언된 public 메서드 중, 반환 타입이 String이고 메서드명이 getUserById이며 파라미터가 Long 타입 1개인 메서드 실행 지점을 정확히 매칭함.
  - 예시: `execution(* net.likelion.bebc25.spring.aop..*Service.*(..))`
  - 설명: net.likelion.bebc25.spring.aop 패키지 및 그 하위 패키지 내에 위치하며 클래스 이름이 Service로 끝나는 모든 클래스의 모든 메서드(반환 타입 및 파라미터 제약 없음) 실행 지점을 매칭함.
  - 예시: `execution(* net.likelion.bebc25.spring.aop.service.*Service.*(..))`
  - 설명: net.likelion.bebc25.spring.aop.service 패키지 내에 직접 위치하며 클래스 이름이 Service로 끝나는 모든 클래스의 모든 메서드 실행 지점을 매칭함. service 패키지의 하위 패키지 내부 클래스들은 매칭 대상에서 제외함.

- @annotation
  특정 어노테이션이 메서드 레벨에 부착된 지점만을 한정 타겟팅하는 지시자. 핵심 비즈니스 로직 소스 코드를 직접 수정하지 않고 공통 관심사를 결합할 때 유용함.
  - 예시: `@annotation(net.likelion.bebc25.spring.aop.annotation.TimeTrace)`
  - 설명: 메서드 정의부에 TimeTrace 어노테이션이 선언된 모든 메서드 실행 지점을 매칭함.

- within
  특정 타입 또는 특정 패키지 하위의 모든 조인포인트를 클래스 수준에서 일괄 매칭하는 지시자. execution과 다르게 메서드 시그니처가 아닌 클래스 타입 범위를 기준으로 필터링함.
  - 예시: `within(net.likelion.bebc25.spring.aop.service..*)`
  - 설명: net.likelion.bebc25.spring.aop.service 패키지 및 그 하위 패키지에 속한 모든 클래스 내부 메서드 실행 지점을 매칭함.
  - 예시: `within(net.likelion.bebc25.spring.aop.service.*)`
  - 설명: net.likelion.bebc25.spring.aop.service 패키지에 직접 속한 모든 클래스의 내부 메서드 실행 지점을 매칭함. service 패키지의 하위 패키지에 속한 클래스들은 매칭 대상에서 제외함.

- bean
  스프링 빈의 ID 또는 빈 이름의 와일드카드 패턴을 기준으로 매칭 대상을 선정하는 스프링 전용 지시자
  - 예시: `bean(*Service)`
  - 설명: 스프링 컨테이너에 등록된 빈의 이름이 Service로 끝나는 모든 스프링 빈의 메서드 호출 지점을 매칭함.

- this 및 target
  this는 런타임에 형성된 AOP 프록시 객체의 타입을 기준으로 매칭하며, target은 프록시가 가리키는 실제 대상 객체의 타입을 기준으로 매칭을 제어함.
  - 예시: `this(net.likelion.bebc25.spring.aop.service.UserService)`
  - 설명: 생성된 프록시 객체가 UserService 인터페이스를 상속 또는 구현한 타입인 경우 매칭함.
  - 예시: `target(net.likelion.bebc25.spring.aop.service.UserServiceImpl)`
  - 설명: 실제 비즈니스 로직을 수행하는 target 객체가 UserServiceImpl 타입인 경우에 매칭을 지원함.

#### 2) 이름을 부여한 포인트컷 (Named Pointcut)
포인트컷 표현식을 직접 매번 기재하지 않고, @Pointcut 어노테이션을 사용하여 의미 있는 이름을 가진 메서드로 정의하는 방식임.

```java
// @Pointcut 어노테이션을 활용해 serviceLayer라는 이름의 포인트컷을 정의함
@Pointcut("within(net.likelion.bebc25.spring.aop.service.*)")
public void serviceLayer() {}
```

#### 3) 포인트컷 표현식 조합
논리 연산자인 AND(&&), OR(||), NOT(!)을 결합하여 복잡한 필터링 조건을 정교하게 구성할 수 있음. 포인트컷 어노테이션(@Pointcut) 내부에 여러 개의 표현식을 직접 조합하여 적용하거나, 이미 정의된 명명된 포인트컷들을 조합함.

- 표현식 직접 조합 적용
  @Pointcut 어노테이션의 값 설정 내에 표현식들을 직접 논리 연산자로 조합하여 정의하는 방식임.

  ```java
  // service 패키지 또는 repository 패키지 내부 클래스의 메서드 실행 지점을 직접 조합함
  @Pointcut("within(net.likelion.bebc25.spring.aop.service.*) || within(net.likelion.bebc25.spring.aop.repository.*)")
  public void serviceOrRepositoryLayer() {}
  ```

- 이름을 부여한 포인트컷 조합
  이미 정의된 포인트컷 메서드 이름들을 논리 연산자로 조합하여 새로운 조합 포인트컷을 정의하는 방식임.

  ```java
  @Pointcut("within(net.likelion.bebc25.spring.aop.service.*)")
  public void serviceLayer() {}

  @Pointcut("within(net.likelion.bebc25.spring.aop.repository.*)")
  public void repositoryLayer() {}

  // 선언한 포인트컷 메서드명들을 연산자로 조합하여 새로운 조합 포인트컷을 정의함
  @Pointcut("serviceLayer() || repositoryLayer()")
  public void serviceOrRepository() {}
  ```

### 6.4.4 Target (대상 객체)
- Aspect를 부여받는 실질적인 비즈니스 핵심 클래스 컴포넌트
- 개발자가 코드상에서 어노테이션을 붙이거나 명시적으로 지정하지 않으며, Pointcut에 의해 필터링된 스프링 빈 객체들이 자동으로 Target이 됨.

### 6.4.5 Advice (공통 기능)
- Pointcut으로 지정된 대상 지점에서 실제로 수행할 구체적인 횡단 관심사(공통 기능)를 구현한 메서드
- 코드상에서 @Before, @Around 등을 적용해 작성하며, 타깃 메서드의 실행 주기에 맞춰 다음과 같이 5가지 동작 시점을 지원함.

#### 1) Advice의 5가지 동작 시점
- @Before (이전): 타깃 메서드가 실행되기 직전에 실행
- @After (이후): 타깃 메서드의 성공 여부와 무관하게 메서드가 종료되면 무조건 실행하는 사후 처리 (try-catch의 finally 블록과 유사함)
- @Around (전후 주변): 타깃 메서드의 호출 전후 과정을 완전히 장악하여 실행 시점 통제, 반환값 조작, 예외 가로채기 등 가장 강력한 제어를 지원함
- @AfterReturning (정상 반환 이후): 타깃 메서드가 예외 발생 없이 성공적으로 실행을 마쳐 결과 값을 반환한 직후에 실행
- @AfterThrowing (예외 발생 이후): 타깃 메서드 실행 도중 예외가 발생하여 예외가 던져지는 시점에 실행

## 6.5 스프링 AOP 적용 방법
스프링 AOP를 활성화하고 적용하기 위해서는 빌드 환경 설정 및 스프링 컨테이너 설정 클래스에서 다음 단계들이 수행되어야 함.

### 6.5.1 빌드 의존성 추가
- 빌드 설정 파일(build.gradle)에 `spring-aspects` 의존성 추가
  - `org.aspectj:aspectjweaver`는 `spring-aspects`의 의존성으로 자동 추가됨

```groovy
implementation 'org.springframework:spring-aspects:7.0.8'
```

### 6.5.2 Aspect 클래스 정의
- 공통 관심사를 담은 클래스를 선언하고 @Aspect 어노테이션을 부여
- 클래스 내부에 적용 대상 범위를 정할 @Pointcut 메서드와 공통 기능을 담은 Advice 메서드를 정의

```java
@Pointcut("execution(* net.likelion.bebc25.spring.aop.springaop.*Driver.*(..))")
private void driverMethods(){ }

@Before("driverMethods()")
public void logBefore(JoinPoint joinPoint){
    System.out.println("[AOP 로그 before] 메서드 실행 전에 처리할 코드를 작성합니다.");
}
```

### 6.5.3 AOP 자동 프록시 가동 설정 추가
- 스프링 설정 클래스(AppConfig) 상단에 @EnableAspectJAutoProxy 어노테이션 추가
- 이 설정이 추가되어야 스프링 컨테이너가 @Aspect 어노테이션이 선언된 빈을 스캔하여 프록시 객체를 자동으로 생성해 줌.

### 6.5.4 Aspect 및 Target 클래스 스프링 빈 등록
- 부가 기능이 구현된 Aspect 클래스와 핵심 비즈니스 로직을 수행할 Target 클래스 모두를 스프링 빈으로 등록
- 스프링 AOP 프록시는 스프링 빈으로 관리되는 객체 간의 메서드 호출에서만 동작함

```java
@Configuration
@EnableAspectJAutoProxy
public class AppConfig {
    @Bean
    public Car car(){
        return new GasolineCar();
    }

    @Bean
    public Driver driver(Car car){
        return new Driver(car);
    }

    @Bean
    public LoggingAspect loggingAspect(){
        return new LoggingAspect();
    }
}
```

# 7. 컴포넌트 스캔과 의존관계 자동 주입

## 7.1 컴포넌트 스캔 (Component Scan)
- 수동으로 @Configuration 클래스 내부에 @Bean 어노테이션을 부여하여 스프링 빈을 일일이 등록하는 방식은 등록할 빈의 개수가 많아질수록 설정 정보가 비대해지고 누락되는 실수가 발생하기 쉬움.
- 스프링은 이러한 설정 파일의 번거로움을 생략하고, 특정 패키지 하위의 클래스들을 검색하여 @Component 어노테이션이 붙은 클래스를 스프링 빈으로 자동 등록해 주는 컴포넌트 스캔 기능을 제공함.
- 스프링 설정 클래스(AppConfig)에 `@ComponentScan(basePackages = "패키지")` 어노테이션을 선언하면 지정한 패키지 범위 내의 클래스들을 검색하여 @Component 어노테이션이 부착된 컴포넌트들을 자동으로 빈으로 등록함.
- @ComponentScan 어노테이션에 패키지 지정을 생략하면 설정 파일이 있는 패키지와 하위 모든 패키지가 컴포넌트 스캔의 대상이 됨
- 빈의 이름은 클래스명의 첫 글자를 소문자로 변환하여 지정됨

```java
@Component
public class GasolineCar implements Car { }
```

```java
@Component
public class HybridCar implements Car { }
```

```java
@Component
public class Driver { }
```

```java
@Configuration
@EnableAspectJAutoProxy
@ComponentScan(basePackages = "net.likelion.bebc25.spring.aop.springaop")
public class AppConfig {
    // 컴포넌트 스캔에 의해 @Component가 부착된 GasolineCar, HybridCar, Driver 등이 자동으로 등록되고 결합되므로 @Bean을 등록할 필요가 없음
}
```

## 7.2 의존관계 자동 주입 (Dependency Auto Injection)
- 각 클래스 내부에서 필요한 의존성 주입을 위해 @Autowired 어노테이션을 사용
- 주입할 매개변수의 타입(인터페이스 또는 클래스)을 기준으로 컨테이너 내의 동일 타입의 빈을 탐색함.
- 타입 검색 결과 동일한 타입의 빈 후보가 여러 개 발견될 경우, 주입 대상을 결정할 수 없어 컨테이너 구동 단계에서 NoUniqueBeanDefinitionException 예외가 발생

### 7.2.1 생성자 주입
- 스프링 빈 클래스의 생성자 상단에 @Autowired를 선언하면 스프링 컨테이너가 생성자 파라미터에 요구되는 빈 객체를 자동으로 매칭하여 주입
- 생성자가 단 1개만 존재할 경우에는 @Autowired 어노테이션을 생략해도 자동으로 의존관계 주입이 처리됨.

```java
@Autowired
public Driver(Car car){
    this.car = car;
}
```

#### 2) 수정자 주입 (Setter Injection)
필드 값을 변경하는 수정자(setter) 메서드에 @Autowired를 선언하여 의존 관계를 주입받는 방식

```java
@Autowired
public void setCar(Car car){
    this.car = car;
}
```

#### 3) 필드 주입 (Field Injection)
자바 클래스의 필드 변수 상단에 @Autowired 어노테이션을 붙여서 스프링 컨테이너가 의존성을 직접 주입하도록 하는 방식

```java
@Autowired
private Car car;
```

### 7.2.2 동일 타입의 다중 빈 자동 주입 제어
동일한 인터페이스(Car)를 구현한 여러 개의 하위 클래스가 스프링 컨테이너에 동시에 빈으로 등록되어 있을 경우, 자동 의존 주입 단계에서 어떤 구현 객체를 매핑하여 사용할지 결정하는 제어 기법
- 동일한 타입의 여러 빈이 있을 경우 예외 발생
  - 타입(Car)만을 기준으로 자동 주입(@Autowired)을 가동하면, 스프링은 대상 후보가 복수 개(GasolineCar, ElectricCar)여서 어떤 빈을 매칭할지 결정하지 못하고 예외(NoUniqueBeanDefinitionException)를 발생시킴.
- 이 충돌 문제를 해결하기 위해 다음 제어 방식을 적용할 수 있음.

#### 1) @Primary 어노테이션 기본값 선언
- 동일한 타입의 빈 중 우선 주입 대상을 기본값으로 선언하는 방법
- 특정 빈 클래스 상단에 @Primary 어노테이션을 설정하면, 호출하는 코드에서 별도로 빈 이름을 기입하지 않더라도 @Primary가 부착된 기본 빈을 최우선적으로 찾아 주입함

```java
@Component
@Primary // GasolineCar가 Car 타입의 Primary가 됨
public class GasolineCar implements Car { }
```

```java
@Component
public class HybridCar implements Car { }
```

#### 2) @Qualifier 어노테이션 지정
- 생성자나 setter 메서드의 매개변수 앞에 @Qualifier 어노테이션을 부착하고 대상 빈의 이름(id)을 문자열로 선언하여 주입 관계를 명시적으로 지정

```java
@Autowired
public Driver(@Qualifier("hybridCar") Car car){
    System.out.println("생성자를 통해 외부에서 Car 구현체를 주입 받음");
    this.car = car;
}
```

#### 3) @Primary와 @Qualifier의 우선순위
- 동일한 타입의 다중 빈 후보 중에서 @Primary와 @Qualifier가 충돌할 때의 처리 기준
- @Primary는 기본 주입 대상을 제공하는 넓은 범위의 설정 방식이고, @Qualifier는 주입 지점에 대상을 구체적으로 지목하는 좁은 범위의 설정 방식
- 스프링은 범위가 좁고 구체적인 설정에 더 높은 우선권을 부여하는 원칙을 따르므로, 상세 지정이 수행된 @Qualifier가 최우선으로 적용되어 주입됨

# 8. 스프링 컨테이너 핵심 메커니즘

## 8.1 스프링 빈 스코프 (Scope)
- 스코프(Scope)는 스프링 빈 객체가 스프링 컨테이너 내부에서 유지되고 유효성을 가지는 생존 범위를 말함

### 8.1.1 싱글톤 스코프 (Singleton Scope)
- 컨테이너 구동 단계에서 빈을 단 하나만 사전 생성하여 전체 사용처에서 하나의 객체를 공유하는 스프링의 기본 스코프 방식
- 상태를 저장하지 않는 무상태 방식으로 빈을 설계해야 멀티 스레드 가동 환경에서 안전한 동작을 보증할 수 있음
- 상태를 공유하지 않고 비즈니스 로직만 공통 수행하는 대부분의 스프링 빈들과 데이터베이스 연결 정보를 가진 DataSource 등 공통 기술 지원 객체 등에 사용

### 8.1.2 프로토타입 스코프 (Prototype Scope)
- 빈 조회 요청이 발생할 때마다 컨테이너가 새로운 빈을 독립적으로 매번 생성하여 반환하는 특수 스코프 방식
- 빈에 @Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE) 어노테이션을 추가
- 스프링 컨테이너는 프로토타입 빈의 생성, 의존 관계 주입, 초기화 콜백까지만 관리하므로, 빈이 사용한 자원의 반납(리소스 해제)은 해당 빈을 제공받은 클라이언트 코드가 직접 책임지고 처리해야 함.
- 생성과 관리의 복잡성 해소를 위해 빈으로 관리하는 것보다 new 연산자를 이용한 직접 생성이 권장됨

```java
@Component
@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE) // "prototype"
public class GasolineCar implements Car { }
```

## 8.2 스프링 빈 생명주기 및 콜백
### 8.2.1 생명주기 콜백의 필요성
- 애플리케이션 시작(구동) 시점에 데이터베이스 커넥션 풀을 사전 연결해 두거나 외부 네트워크 소켓 세션을 연결하는 작업, 그리고 서버가 정지(종료)되기 직전에 열려 있던 커넥션을 부드럽게 닫고(Graceful Shutdown) 메모리 버퍼 자원을 안전하게 반환하는 등 외부 리소스를 연동 제어할 때 사용함
- 생성자는 객체 내부의 구조를 채우고 메모리를 할당하는 최소한의 책임(생성 자체)에만 집중해야 함
- 따라서 데이터베이스 커넥션 풀 사전 연결이나 네트워크 소켓 연결 등의 무거운 외부 리소스 작업은 생성자에서 수행하지 않고, 객체 생성이 완전히 끝나고 의존성 주입이 보장된 이후 호출되는 초기화 콜백 단계에서 실행해야 함
- 특히 필드 주입(Field Injection)이나 수정자 주입(Setter Injection)을 사용하는 경우, 생성자가 호출되는 시점에는 주입받을 빈들이 아직 null 상태이므로 생성자 내에서 의존 객체를 활용한 초기화 코드를 실행하면 NullPointerException 예외가 발생함. 따라서 이 주입 방식을 사용할 때는 의존 관계 연동을 위한 초기화 메서드의 활용이 필수적

### 8.2.2 스프링 빈 생명주기 전체적인 흐름
- 1. 스프링 컨테이너 기동
- 2. 스프링 빈 인스턴스 생성
- 3. 의존 관계 주입 완료
- 4. 초기화 메서드 수행(@PostConstruct)
- 5. 빈의 기동 및 비즈니스 로직 수행
- 6. 소멸 메서드 수행(@PreDestroy)
- 7. 스프링 컨테이너 종료 및 리소스 소멸

### 8.2.3 빈의 생명주기 콜백 지정 방식
- 어노테이션 지정 방식
  - 빈 내부의 실행 메서드 상단에 @PostConstruct(초기화 메서드) 및 @PreDestroy(소멸 메서드) 자바 표준 어노테이션을 지정
  - Jakarta Annotations API 라이브러리 필요
  - 자바 표준 어노테이션이므로 스프링에 종속되지 않아서 사용을 권장하는 방식
- 인터페이스 상속 방식
  - InitializingBean(afterPropertiesSet 구현) 및 DisposableBean(destroy 구현) 인터페이스 직접 구현을 통한 제어
  - 스프링 프레임워크 기술 규격에 코드가 종속되므로 사용을 권장하지 않음
- 빈 설정 옵션 방식
  - 자바 빈 설정 작성 시 @Bean(initMethod = "메서드명", destroyMethod = "메서드명") 형태로 콜백 이름 기입 및 지정

```java
import jakarta.annotation.PostConstruct;
import jakarta.annotation.PreDestroy;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;

@Component
public class TempFileSupport {
    @Value("resoueces/temp.log") // 단순한 값을 주입할 때 사용
    private String tempFilePath;

    public TempFileSupport() {
        System.out.println("생성자 호출: 객체가 생성됨. 임시파일경로 = " + tempFilePath);
    }

    // 1. 초기화 콜백 (의존 관계 주입이 끝난 직후 가동되어 임시 작업 파일 생성)
    @PostConstruct
    public void initTempFile() {
        System.out.println("초기화 콜백: 작업용 임시 파일 생성 완료. 경로 = " + tempFilePath);
    }

    // 2. 비즈니스 기능 실행
    public void writeLog(String logMessage) {
        System.out.println("임시 파일에 로그 기록: " + logMessage);
    }

    // 3. 소멸 콜백 (컨테이너 종료 직전에 가동되어 자원 해제 및 파일 자동 삭제)
    @PreDestroy
    public void clearTempFile() {
        System.out.println("소멸 콜백: 임시 파일 안전하게 삭제 및 디스크 자원 반환.");
    }
}
```

- 설정 클래스 등록 (AppConfig):
```java
@Configuration
@ComponentScan
public class AppConfig { }
```

- 메인 구동 및 로그 검증 (SpringCoreApplication):
```java
public class SpringCoreApplication {
    public static void main(String[] args) {
        // 1. 컨테이너 기동 및 설정 정보 로딩 (초기화 콜백 자동 발동)
        AnnotationConfigApplicationContext context = 
                new AnnotationConfigApplicationContext(AppConfig.class);

        // 2. 빈 조회 및 비즈니스 사용
        TempFileSupport support = context.getBean(TempFileSupport.class);
        support.writeLog("사용자 로그인 작업 수행 로그 기록.");

        // 3. 스프링 컨테이너 종료
        context.close();
    }
}
```

- 콘솔 출력 로그
```text
생성자 호출: 객체가 생성됨. 임시파일경로 = null
초기화 콜백: 작업용 임시 파일 생성 완료. 경로 = resoueces/temp.log
임시 파일에 로그 기록: 사용자 로그인 작업 수행 로그 기록.
소멸 콜백: 임시 파일 안전하게 삭제 및 디스크 자원 반환.
```
