# 1. 스프링 부트 개요

## 1.1 스프링 부트의 정의와 역할
- 기존 스프링 프레임워크의 번거로운 환경 설정이나 객체를 복잡하게 등록하는 과정을 알아서 자동으로 해결해 주고 필요한 라이브러리들의 버전 충돌까지 스스로 관리해 주어, 결과적으로 스프링 개발을 훨씬 쉽고 편리하게 진행할 수 있도록 지원해 주는 프레임워크
- 웹 서버(Tomcat 등)를 내장하고 있어 별도의 WAS(Web Application Server) 설치 없이 단독 실행 가능한 애플리케이션 빌드를 지원함.

## 1.2 스프링 프레임워크와 스프링 부트의 차이점
### 1.2.1 라이브러리 버전 조율
- 스프링 프레임워크
    - 연동하려는 라이브러리 간 버전 충돌을 방지하기 위해 개발자가 하나씩 버전을 지정하고 호환성을 직접 테스트해야 함.
- 스프링 부트
    - 특정 개발 목적(웹 개발, 데이터베이스 연동, 보안 설정 등)에 필요한 관련 라이브러리 목록을 한 묶음으로 묶어낸 스타터(Starter) 의존성 패키지를 제공함.
    - 개발자가 웹 개발용 스타터(spring-boot-starter-web) 하나만 추가해 두면, 내부적으로 호환되는 Spring Web MVC, Tomcat WAS, Jackson JSON 변환기 등 관련 라이브러리가 검증된 최적 버전 세트로 한 번에 도입되어 의존성 충돌과 복잡한 세부 버전 설정을 방지함.

### 1.2.2 웹 서버(WAS) 설정 및 배포 방식
- 스프링 프레임워크
    - Tomcat 등 외부 WAS(Web Application Server)를 별도 설치하고, 프로젝트를 WAR 파일로 빌드하여 WAS에 배포 및 실행해야 함.
- 스프링 부트
    - 내장 Tomcat 서버를 탑재하여 단독 실행 가능한 JAR 파일로 컴파일 및 자동 빌드하여 실행을 편리하게 함.

# 2. 스프링 부트 프로젝트 환경 구축

## 2.1 스프링 이니셜라이저로 프로젝트 생성
- https://start.spring.io
- 스프링 부트 프로젝트의 초기 뼈대 구조와 설정 파일을 웹 상에서 빠르고 편리하게 구성하여 다운로드할 수 있도록 돕는 공식 웹 서비스
- 프로젝트에 사용할 빌드 도구(Gradle, Maven), 개발 언어(Java, Kotlin), 스프링 부트 버전, 자바 버전, 패키징 타입 및 필요한 핵심 외부 라이브러리(Dependencies) 의존성을 사전에 간편하게 구성할 수 있음.

### 2.1.2 이니셜라이저 접속
스프링 이니셜라이저 웹 사이트(https://start.spring.io) 접속

### 2.1.3 프로젝트 기본 빌드 환경 설정
프로젝트 빌드 환경을 설계 목적에 맞춰 다음과 같이 선택하고 설정
- Project: Gradle - Groovy
- Language: Java
- Spring Boot: 4.1.0 (SNAPSHOT 또는 M 등의 미리보기 라벨이 붙지 않은 안정된 최신 정식 버전 선택)
- Project Metadata:
    - Group: 회사명 또는 단체 도메인명 (`net.likelion.bebc25`)
    - Artifact: 프로젝트명 (`spring-boot-initializr`)
    - Package name: `net.likelion.bebc25.initializr`
    - Packaging: Jar (스프링 부트 애플리케이션의 단독 실행 환경 구축을 용이하게 하므로 Jar 선택)
        + JAR (Java Archive): 내장 WAS(Tomcat 등)를 포함하여 단독으로 실행(java -jar) 가능한 자바 애플리케이션 압축 포맷
        + WAR (Web Application Archive): 웹 구성 요소(JSP, 서블릿 등)를 포함하여 별도의 외장 WAS가 반드시 설치된 서버 환경에 배포 및 실행해야 하는 웹 애플리케이션 압축 포맷
    - Java: 로컬 PC에 설치된 JDK 버전에 맞게 설정 (25)

### 2.1.4 의존성 추가
- Dependencies 영역 우측에 있는 `ADD DEPENDENCIES...` 버튼을 클릭하여 프로젝트 실행에 필요한 스타터 라이브러리들을 선택함.
- 의존성을 전혀 선택하지 않아도 모든 스타터의 모태가 되는 기본 스타터(`spring-boot-starter`)가 자동으로 추가되어 스프링의 기본 동작이 가능함.

#### 1) 일반 라이브러리
- 개발자가 필요한 특정 기능의 JAR 라이브러리를 직접 하나씩 찾아 명시해야 함
- 라이브러리 간 연동 시 호환 버전까지 수동으로 조율해야 하므로 의존성 버전 충돌이 발생하기 쉬움

#### 2) 스프링 부트 스타터 라이브러리
- 특정 목적(웹 개발, DB 연동 등)에 맞춰 상호 검증된 호환 라이브러리들을 한 묶음으로 패키징하여 제공하는 의존성 조합 규격
- 스타터 하나만 가져오면 연관된 수십 개의 하부 의존 라이브러리와 적합한 버전 정보가 Gradle에 의해 한 번에 자동 로딩되어 버전 관리 복잡성을 대폭 낮춤

- 스타터 라이브러리의 네이밍 규칙
    - 스프링 공식 스타터: `spring-boot-starter-기술명` 형식으로 명명하며, 스프링 공식 개발 팀에서 품질을 직접 보증하고 묶어 제공하는 스타터 (예: `spring-boot-starter-web`, `spring-boot-starter-security`)
    - 서드파티 및 오픈소스 스타터: `기술명-spring-boot-starter` 형식으로 명명하며, 외부 벤더사나 커뮤니티 진영에서 자사의 모듈을 스프링 부트에 맞게 조립하여 개별 제공하는 스타터 (예: `mybatis-spring-boot-starter`)

#### 3) 대표적인 라이브러리와 스타터
- Spring Boot Starter (`spring-boot-starter`): 프로젝트 생성 시 기본 탑재되는 최상위 스타터로 스프링 코어 컨텍스트, DI 컨테이너, 로깅 모듈 등을 내장함.
- Spring Web (`spring-boot-starter-web`): 웹 애플리케이션 개발을 지원하는 스타터 (Tomcat 서버 내장, 웹 개발에 필요한 라이브러리 포함).
- Lombok (`org.projectlombok:lombok`): getter, setter 등 반복적인 표준 코드 자동 완성 지원.
- Spring Data JDBC (`spring-boot-starter-data-jdbc`): 가벼운 데이터베이스 연동 지원 (단순 SQL 매핑 방식).
- H2 Database (`com.h2database:h2`): 개발 및 테스트용 인메모리 경량 관계형 데이터베이스 지원.
- Spring Data JPA (`spring-boot-starter-data-jpa`): ORM 기술 표준인 JPA를 활용한 고도화된 데이터베이스 영속성 제어 지원.
- MySQL Driver (`mysql-connector-j`): 상용 서비스 개발 시 주로 사용하는 MySQL 관계형 데이터베이스 연동 드라이버 제공.
- Thymeleaf (`spring-boot-starter-thymeleaf`): 서버 사이드 HTML 뷰 템플릿 동적 렌더링 지원.
- MyBatis Framework (`mybatis-spring-boot-starter`): 객체와 SQL 간 매핑을 통해 편리한 데이터베이스 영속성 제어 지원.
- Spring Security (`spring-boot-starter-security`): 세션 기반 사용자 인증 및 권한 부여 등 웹 보안 설정 제어.

# 3. 스프링 부트 빌드 설정

## 3.1 build.gradle 설정
- 공식 문서: https://docs.gradle.org

### 3.1.1 스프링 부트의 기본 build.gradle
```groovy
plugins {
    id 'java'
    id 'org.springframework.boot' version '4.1.0'
    id 'io.spring.dependency-management' version '1.1.7'
}

group = 'net.likelion.bebc25'
version = '0.0.1-SNAPSHOT'

java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(25)
    }
}

repositories {
    mavenCentral()
}

dependencies {
    implementation 'org.springframework.boot:spring-boot-starter'
    implementation 'org.springframework.boot:spring-boot-starter-web'
    compileOnly 'org.projectlombok:lombok'
    runtimeOnly 'com.mysql:mysql-connector-j'
    annotationProcessor 'org.projectlombok:lombok'
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
    testRuntimeOnly 'org.junit.platform:junit-platform-launcher'
}

tasks.named('test') {
    useJUnitPlatform()
}
```

### 3.1.2 플러그인 설정 (plugins)
- 플러그인은 Gradle 빌드 시스템에 추가 기능을 제공하는 재사용 가능한 소프트웨어로 빌드에 특별한 작업이나 설정을 추가할 수 있음.
- Gradle에 기본으로 포함된 코어 플러그인, Gradle 커뮤니티에서 배포한 플러그인, 사용자 정의 플러그인을 추가할 수 있음.
    - [Gradle plugin 포털](https://plugins.gradle.org/)에서 커뮤니티 플러그인을 검색하고 상세 정보를 확인할 수 있음.
    - lombok 플러그인: Lombok 의존성 구성을 자동화 해주므로 dependencies에 다음과 같은 의존성 구성을 추가할 필요가 없음.
      ```groovy
      // 컴파일 시점에 롬복 어노테이션 API 사용
      compileOnly 'org.projectlombok:lombok'
      // 컴파일 빌드 단계에서 롬복 어노테이션 프로세서를 구동하여 코드 자동 생성
      annotationProcessor 'org.projectlombok:lombok'
      // 테스트 컴파일 환경에서도 롬복 어노테이션 사용할 경우 추가
      testCompileOnly 'org.projectlombok:lombok'
      // 테스트 컴파일 단계에서 프로세서 구동
      testAnnotationProcessor 'org.projectlombok:lombok'
      ```
```groovy
plugins {
    id 'java'
    id 'org.springframework.boot' version '4.1.0'
    id 'io.spring.dependency-management' version '1.1.7'
    // https://plugins.gradle.org/plugin/io.freefair.lombok
    id 'io.freefair.lombok' version '9.5.0'
}
```

- java: Gradle이 자바 프로젝트를 인식하여 컴파일 및 빌드할 수 있는 표준 자바 기능 적용.
- org.springframework.boot: 스프링 부트 애플리케이션의 빌드 및 패키징(단독 실행 가능한 JAR 생성)과 실행(bootRun 태스크) 기능을 지원하며, 의존성 버전 관리의 기준 정보 제공.
- io.spring.dependency-management: 스프링 부트가 정의한 버전 정보를 바탕으로 외부 라이브러리 추가 시 버전 명시를 생략해도 적절한 호환 버전을 자동으로 할당하여 관리.
- io.freefair.lombok: 컴파일 단계에서 반복적인 코드를 자동 생성해 주는 롬복 라이브러리 의존성 및 어노테이션 프로세서 설정을 Gradle 빌드 환경에 통합하여 자동 적용.

### 3.1.3 프로젝트 식별 설정 (group, version)
빌드된 산출물(JAR 등)을 배포하거나 공유할 때 사용하는 라이브러리의 고유 식별 정보.
```groovy
group = 'net.likelion.bebc25'
version = '0.0.1-SNAPSHOT'
description = 'spring-boot-intellij'
```

- group: 프로젝트 배포 시 다른 라이브러리와 식별하기 위해 단체의 도메인 기반 고유 그룹명 지정.
- version: 애플리케이션의 개발 버전을 명시하며, 개발 중인 단계는 주로 SNAPSHOT 명칭 부여.
- description: 프로젝트의 성격이나 목적을 나타내는 간략한 설명을 명시하며, 빌드 결과물이나 외부 시스템에서 프로젝트 정보를 참조할 때 활용.

### 3.1.4 자바 툴체인 설정 (java.toolchain)
로컬 PC의 환경 변수(JAVA_HOME)와 상관없이 빌드와 컴파일 및 실행에 사용할 JDK 버전을 일관되게 강제하는 기능.
```groovy
java {
  toolchain {
    languageVersion = JavaLanguageVersion.of(25)
  }
}
```

- languageVersion: 프로젝트가 동작할 JVM 런타임을 JDK 25 버전으로 강제하며, 만약 현재 PC에 없으면 Gradle이 원격 저장소에서 해당 버전을 자동으로 다운로드하여 빌드에 영구적으로 캐싱하여 사용.

### 3.1.5 라이브러리 저장소 설정 (repositories)
프로젝트가 사용하는 외부 의존 라이브러리 파일들을 네트워크를 통해 받아올 온라인 서버 저장소 선언.
```groovy
repositories {
  mavenCentral()
}
```

- mavenCentral(): 메이븐 중앙 저장소를 라이브러리 다운로드 경로로 지정.

### 3.1.6 의존성 구성 키워드 (dependencies)
- 컴파일, 테스트, 실행 시 활용할 라이브러리들을 Gradle이 다운로드하도록 지정.
```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter'
    compileOnly 'org.projectlombok:lombok'
    runtimeOnly 'com.mysql:mysql-connector-j'
    developmentOnly 'org.springframework.boot:spring-boot-devtools'
    annotationProcessor 'org.projectlombok:lombok'
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
    testCompileOnly 'org.projectlombok:lombok'
    testAnnotationProcessor 'org.projectlombok:lombok'
    testRuntimeOnly 'org.junit.platform:junit-platform-launcher'
}
```

#### 1) 의존성 구성 (Dependency Configuration)
- 동작 시점별 구분
    - 컴파일 단계: compileOnly, testCompileOnly
    - 런타임 단계: runtimeOnly, testRuntimeOnly
    - 컴파일 및 런타임 공통: implementation, testImplementation
    - 어노테이션 프로세스: annotationProcessor, testAnnotationProcessor
    - 개발 단계 전용: developmentOnly
- 소스 영역별 구분
    - 메인 영역 (src/main): implementation, compileOnly, runtimeOnly, annotationProcessor, developmentOnly
    - 테스트 영역 (src/test): testImplementation, testCompileOnly, testRuntimeOnly, testAnnotationProcessor

#### 2) 메인 소스코드 영역에 적용되는 속성 (src/main)
- compileOnly: 컴파일에만 적용. 자바 소스 컴파일 시점에만 사용되고, 실제 프로그램 실행(런타임) 단계에는 제외되므로 최종 실행 파일(JAR)의 용량을 줄이는 설정.
    - lombok: 코드 내 반복해서 적어야 하는 표준 메서드(getter, setter 등) 작성을 어노테이션만으로 자동 완성하여 개발 생산성을 높이는 도구로 컴파일 시점에만 사용됨.

- runtimeOnly: 런타임에만 적용. 소스코드 컴파일 시점에는 참조할 수 없으나, 실제 애플리케이션 작동(런타임) 단계에 동적으로 로드되어 구동을 보조하는 라이브러리 지정.
    - mysql-connector-j: 자바 프로그램과 MySQL 데이터베이스를 통신 인터페이스로 연결해 주는 JDBC 드라이버 라이브러리.

- implementation: 컴파일과 런타임 모두 적용. 메인 소스 영역의 컴파일과 실제 실행(런타임) 모두에 공통으로 사용되며, 테스트 소스 영역으로 자동 상속되어 테스트 수행 시에도 제공되는 가장 일반적인 라이브러리 지정.
    - spring-boot-starter: 프로젝트 생성 시 기본 탑재되는 최상위 스타터로 스프링 코어 컨텍스트, DI 컨테이너, 로깅 모듈 등을 내장함.

- annotationProcessor: 특수 컴파일 설정. 컴파일 단계에서 자바 어노테이션 정보들을 스캔하여 실질적인 클래스 및 바이트코드로 자동 조립하고 변환해 주는 전용 컴파일 엔진.
    - lombok: 롬복 어노테이션을 분석하여 실제 컴파일된 클래스 파일 내에 구현 코드를 주입하는 빌드 단계 플러그인.

- developmentOnly: 개발 단계에만 적용. 로컬 개발 환경에서 유용한 기능을 제공하며, 실제 운영 환경으로 배포하기 위해 JAR 파일 등으로 패키징할 때는 자동으로 배포 대상에서 제외되는 설정.
    - spring-boot-devtools: 소스코드 변경 시 자동으로 애플리케이션을 재기동하고 웹 브라우저 캐싱을 방지하는 등 개발 생산성을 높여주는 개발 지원 도구.

#### 2) 테스트 소스코드 영역에 적용되는 속성 (src/test)
- testCompileOnly: 테스트 컴파일에만 적용. 테스트 소스 컴파일 시점에만 사용되고 테스트 실행 시에는 로드되지 않는 테스트 전용 컴파일 설정.
    - lombok: 테스트 코드 내부에서도 게터, 세터 및 생성자 코드를 자동 완성하여 테스트 작성을 돕는 개발 생산성 도구.

- testRuntimeOnly: 런타임에만 적용. 테스트 코드를 작성하고 컴파일할 때는 쓰이지 않고, 테스트가 실제로 작동하는 테스트 런타임 단계에만 사용되어 구동을 보조하는 엔진 설정.
    - junit-platform-launcher: 인텔리제이 등의 개발 도구나 Gradle이 작성된 JUnit 5 테스트를 정상적으로 인식하여 구동하고 그 결과를 수집할 수 있게 돕는 테스트 런처 엔진.

- testImplementation: 컴파일과 런타임 모두 적용. 테스트 소스 영역의 컴파일과 실행 모두에 공통 사용되며, 실제 배포용 메인 소스 영역에는 포함되지 않도록 완벽하게 차단되는 검증 라이브러리 지정.
    - spring-boot-starter-test: JUnit 5, Mockito, AssertJ 등 스프링 부트 애플리케이션 테스트에 필요한 검증 라이브러리를 내장함.

- testAnnotationProcessor: 테스트 특수 컴파일 설정. 테스트 클래스 컴파일 시 어노테이션을 스캔하여 바이트코드로 가공해 주는 테스트 전용 컴파일 엔진.
    - lombok: 테스트 컴파일 시점에 롬복 어노테이션을 파싱하여 코드를 자동 삽입해 주는 테스트용 컴파일 플러그인.

> compileOnly는 testCompileOnly에 전파, runtimeOnly는 testRuntimeOnly에 전파, implementation은 testImplementation에 전파되므로 테스트 영역에 따라 작성하지 않아도 됨.

### 3.1.7 태스크 제어 설정 (tasks.named)
- Gradle이 기본 제공하는 작업을 가로채어 세부적인 작업 속성을 변경하는 설정.
```groovy
tasks.named('test') {
  useJUnitPlatform()
}
```

- useJUnitPlatform(): 테스트 실행 태스크를 타깃으로 삼아, 자바의 표준 테스트 엔진인 JUnit 5를 활용해 작동하도록 지정.

# 4. 스프링 부트의 자동 빈 등록 메커니즘

## 4.1 메인 실행 클래스와 @SpringBootApplication
- 스프링 부트 애플리케이션의 시작점인 Application 클래스 위에 @SpringBootApplication 어노테이션을 부여하여 스프링 컨테이너를 구동
- 이 단일 어노테이션은 스프링 부트 자동화의 핵심인 다음 3대 어노테이션을 포함하고 있음
    - @SpringBootConfiguration: 스프링 부트 전용 설정 클래스임을 명시하며 내부적으로 @Configuration 기능을 공유함
    - @EnableAutoConfiguration: build.gradle에 추가한 라이브러리를 자동으로 감지하여, 각 라이브러리에 맞는 스프링 빈 설정을 개발자가 직접 코드로 작성하지 않아도 자동으로 완성해 줌.
        - spring-boot-starter-web을 추가하면 내장 톰캣 서버와 Spring MVC에 필요한 빈들을 자동으로 등록함
        - spring-aspects 의존성만 build.gradle에 추가해 두면 자동으로 Spring AOP가 동작함
    - @ComponentScan: 컴포넌트 탐색을 수행함. 이 설정 클래스가 위치한 패키지를 루트 패키지로 삼아 하위 모든 패키지의 @Component 계열 빈들을 스캔함

## 4.2 자동 의존성 주입 예시

```java
package net.likelion.bebc25.intellij;

public interface Car {
    void startEngine();
    void drive();
    void stopEngine();
}
```

```java
package net.likelion.bebc25.intellij;

import org.springframework.stereotype.Component;

@Component
public class GasolineCar implements Car {
    public void startEngine() {
        System.out.println("가솔린 엔진에 점화하여 시동을 켭니다.");
    }
    public void drive() {
        System.out.println("가솔린 동력으로 주행합니다.");
    }
    public void stopEngine() {
        System.out.println("가솔린 공급을 차단하여 시동을 끕니다.");
    }
}
```

```java
package net.likelion.bebc25.intellij;

import org.springframework.stereotype.Component;

@Component
public class Driver {
    private final Car car;

    // 단일 생성자이므로 @Autowired 어노테이션 생략 가능
    public Driver(Car car) {
        this.car = car;
    }

    public void driveCar() {
        car.startEngine();
        car.drive();
        car.stopEngine();
    }
}
```

# 5. 스프링 부트 설정 파일과 프로파일

## 5.1 Properties와 YAML 설정 포맷 비교
스프링 부트 애플리케이션의 전반적인 설정을 정의할 때 사용하는 application.properties와 application.yml의 문법 및 구조적 특징

- Properties 설정 (`application.properties`)
    - 구조: 평면적인 key=value 구조
    - 장점: 작성법이 단순하여 들여쓰기 실수로 인한 에러가 없음
    - 단점: 중복되는 설정 키명을 매 줄마다 입력해야 해서 비대해짐
  ```properties
  server.port=8080
  spring.application.name=spring-boot-intellij
  ```

- YAML 설정 (`application.yml`)
    - 구조: 들여쓰기 공백 기준의 계층형 트리 구조
    - 장점: 접두사의 중복 작성을 배제하여 구조가 간결하고 가독성이 뛰어남
    - 단점: 공백 1칸의 정렬 오타에도 로드 실패 예외를 유발함

  ```yaml
  server:
    port: 8080
  spring:
    application:
      name: spring-boot-intellij
  ```

## 5.2 스프링 부트의 주요 설정
- https://docs.spring.io/spring-boot/appendix/application-properties/index.html 참고
- 스프링 부트는 아무런 설정을 하지 않아도 정해진 규칙에 따라 기본값으로 동작함.
- 커스터마이징이 필요한 경우 application.properties 또는 application.yml에 해당 키를 명시하여 기본값을 덮어씀.

```yaml
# ─── 서버 설정 ─────────────────────────────────────────────
server:
  # 내장 웹 서버(톰캣)가 HTTP 요청을 수신할 포트 번호
  port: 8080
  servlet:
    # 모든 URL 요청의 앞에 공통으로 붙는 컨텍스트 루트 경로
    context-path: /

# ─── 스프링 핵심 설정 ──────────────────────────────────────
spring:
  # ─── 애플리케이션 정보 ──────────────────────────────────────
  application:
    # 애플리케이션의 고유 이름으로, 모니터링 도구나 로그에서 서비스 식별에 활용
    name: 

  # ─── 데이터소스 연결 ────────────────────────────────────────
  datasource:
    # 연결할 데이터베이스 서버의 JDBC URL (DB 종류, 호스트, 포트, DB명 포함)
    url:
    # 데이터베이스 접속 계정명
    username:
    # 데이터베이스 접속 비밀번호
    password:
    # JDBC 드라이버 클래스명 (생략 시 URL에서 자동 감지)
    driver-class-name:

  # ─── JPA / Hibernate 설정 ──────────────────────────────────
  jpa:
    hibernate:
      # 애플리케이션 기동 시 DB 테이블 자동 생성·삭제 전략
```

## 5.3 프로파일(Profile)을 활용한 배포 환경 격리
- 프로파일은 구동 환경에 따라 데이터베이스 url이나 포트 등 완전히 격리된 설정을 적용하기 위한 환경 세트 분류 장치

### 5.3.1 배포 환경 유형
#### 1) local
- 로컬 개발 환경
- 개발자의 개인 PC에서 구동하는 환경
- 로컬 DB를 연결하거나 H2 인메모리 DB를 사용하며, 디버깅 및 빠른 확인을 위해 상세 로그와 DevTools를 활성화함

#### 2) staging
- 스테이징 환경
- 실운영 배포 직전에 최종 검증을 수행하는 테스트 전용 서버 환경
- 운영과 동일한 인프라 구성을 유지하지만 실제 사용자 트래픽은 유입되지 않는 환경

#### 3) prod
- 실운영 환경
- 실제 서비스가 제공되는 라이브 서버 환경
- 보안 설정을 강화하고 운영 DB에 연결하며, 불필요한 로그 출력을 최소화함

### 5.3.2 환경별 다른 설정 파일 적용
- 설정 파일명에 `application-{profile}.properties` 형식의 이름을 지정하여 각 환경 설정을 분리함
    - 로컬: `application-local.properties`
    - 스테이징: `application-staging.properties`
    - 운영: `application-prod.properties`

- 활성화 지정 방법
    - application.properties의 spring.profiles.active 값 지정
      ```properties
      spring.profiles.active=local
      ```
    - 운영 체제의 환경 변수에 spring.profiles.active 값 지정
      ```bash
      # Linux / macOS
      export SPRING_PROFILES_ACTIVE=prod
  
      # Windows (명령 프롬프트)
      set SPRING_PROFILES_ACTIVE=prod
      ```
    - 서버 실행 시 JVM 실행 인수(`-D`) 지정
      ```bash
      java -Dspring.profiles.active=prod -jar app.jar
      ```

- 프로파일 탐색 방식
    - `spring.profiles.active`에 지정한 값이 그대로 프로파일명이 되며, 스프링 부트는 `application-{프로파일명}.properties` 패턴으로 파일을 탐색함
    - 기동 시 로드 순서
        1. `application.properties` 로드 (모든 환경에 공통 적용되는 기본 설정)
        2. `spring.profiles.active` 값 확인 (예: `local`)
        3. `application-local.properties` 로드
        4. 프로파일 전용 설정이 기본 설정과 중복되는 키를 덮어씌움

## 5.4 @Profile 어노테이션을 활용한 조건부 빈 등록
- 특정 프로파일이 활성화된 환경에서만 스프링 컨테이너에 등록되는 특수 빈을 정의할 때 사용함
- 실무에서는 로컬 개발 단계의 모조(Mock) 결제 수단 빈과 실서버에서 가동할 실제 카드 결제 API 빈을 안전하게 격리 등록할 때 활용함

```java
public interface PaymentService {
    void processPayment(int amount);
}
```

```java
import org.springframework.context.annotation.Profile;
import org.springframework.stereotype.Component;

@Component
@Profile("local") // local 프로파일 활성화 시에만 컨테이너에 등록됨
public class FakePaymentService implements PaymentService {
    public void processPayment(int amount) {
        System.out.println("임시 개발용 결제: " + amount + "원 가상 승인 완료.");
    }
}
```

```java
import org.springframework.context.annotation.Profile;
import org.springframework.stereotype.Component;

@Component
@Profile("prod") // prod 프로파일 활성화 시에만 컨테이너에 등록됨
public class RealPaymentService implements PaymentService {
    public void processPayment(int amount) {
        System.out.println("실제 금융 카드 결제 API 호출: " + amount + "원 승인 요청 진행.");
    }
}
```

```java
import org.springframework.stereotype.Component;

@Component
public class OrderService {
    private final PaymentService paymentService;

    // 활성 프로파일에 따라 FakePaymentService 또는 RealPaymentService가 자동 주입됨
    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }

    public void placeOrder(int amount) {
        paymentService.processPayment(amount);
    }
}
```

# 6. 로깅(Logging) 시스템

## 6.1 System.out.println의 한계와 로깅 프레임워크
- System.out.println을 사용하면 콘솔에 텍스트만 표시되므로 로그 레벨별로 통제가 불가능하고, 동기적 입출력 처리로 인해 애플리케이션의 연산 처리 속도를 심각하게 떨어트려 실무 운영 서버 배포 시 사용을 하지 않음
- 로깅 프레임워크는 출력 레벨 통제, 파일, DB 및 원격 로깅 저장, 성능 최적화 등을 지원함
- 스프링 부트는 로깅 API 표준 인터페이스인 SLF4J를 기본 지원하고, 실제 로그를 기록하는 내부 구현체로 Logback 라이브러리를 바인딩하여 기동함(spring-boot-starter의 의존성으로 인해 자동으로 추가됨)

## 6.2 롬복(Lombok) @Slf4j를 활용한 로그 기록
- 롬복의 @Slf4j 어노테이션을 클래스 상단에 붙여두면 자동으로 static final Logger 변수가 컴파일 단계에서 생성되어, 단순 호출만으로 정교한 로그를 기록할 수 있음

```java
@Slf4j
public class LoggingAspect { }
```

```java
private static final Logger log = LoggerFactory.getLogger(LoggingAspect.class); // 자동 추가됨
```

## 6.3 패키지별 로그 레벨 제어 설정
- 로그 레벨은 중요도에 따라 TRACE - DEBUG - INFO - WARN - ERROR 순으로 로깅 수준이 높아짐
    - 로그 레벨을 DEBUG로 설정할 경우 DEBUG, INFO, WARN, ERROR 레벨의 로그가 출력됨.
    - 로그 레벨을 WAR으로 설정할 경우 WARN, ERROR 레벨의 로그가 출력됨.
- 스프링 부트의 기본 로그 출력 레벨(Default Level)은 INFO로 지정되어 있음.
- 로그 레벨을 지정하지 않으면 INFO 이상의 상위 레벨(INFO, WARN, ERROR) 로그만 화면에 노출되고 하위 레벨(DEBUG, TRACE)은 숨겨짐
- application.properties에서 특정 패키지를 지정하여 해당 패키지의 로그 필터 레벨을 TRACE나 DEBUG로 낮춤으로써 TRACE나 DEBUG 레벨의 로그를 출력할 수 있음.

```properties
# 기본값
logging.level.root=INFO
# 지정한 패키지 하위의 로그는 DEBUG 단계까지 모두 콘솔에 노출함
logging.level.net.likelion.bebc25=DEBUG
```

## 6.4 파일 로그 저장 설정
- 스프링 부트는 기본적으로 콘솔에만 로그를 출력하며, 파일에 로그를 저장하려면 application.properties에 저장 경로를 명시해야 함
- 파일 크기가 10MB가 되면 자동으로 롤링(기존 파일을 번호를 붙여 보관하고 새 파일 생성)이 수행됨

```properties
# 방법 1: 파일명 직접 지정 (프로젝트 루트 기준 상대 경로)
logging.file.name=logs/app.log

# 방법 2: 디렉토리만 지정 (해당 경로에 spring.log 파일이 자동 생성됨)
logging.file.path=logs

# 롤링 파일 크기 지정 (기본값: 10MB, 초과 시 기존 파일을 번호를 붙여 보관하고 새 파일 생성)
logging.logback.rollingpolicy.max-file-size=50MB
# 보관할 롤링 파일의 최대 개수 (기본값: 7, 초과 시 가장 오래된 파일부터 자동 삭제)
logging.logback.rollingpolicy.max-history=30
```

- logging.file.name과 logging.file.path를 동시에 지정하면 logging.file.name이 우선 적용됨

# 7. 스프링 통합 테스트

## 7.1 통합 테스트 (Integration Test)
- 통합 테스트: 스프링 컨테이너 내 여러 빈들의 실제 결합 및 작동 상태를 전체적으로 가동해 검증하는 테스트
- @SpringBootTest: 실제 운영 애플리케이션 환경과 유사하게 스프링 컨테이너를 구동하여 테스트 대상 빈들을 주입받아 제어할 수 있도록 돕는 테스트용 핵심 어노테이션
- spring-boot-starter-test 의존성 하나에 통합 테스트에 필요한 핵심 라이브러리가 모두 내장되어 있으므로 별도 추가 없이 바로 테스트 작성이 가능함
    - JUnit 5 (Jupiter): 테스트 메서드 작성 및 실행 프레임워크 (@Test, @BeforeEach 등)
    - Spring Test: @SpringBootTest로 스프링 컨테이너를 가동하는 테스트 지원 모듈
    - AssertJ: assertThat() 체이닝 방식의 가독성 높은 검증 라이브러리
    - Mockito: 모의 객체(Mock)를 생성하여 의존성을 가짜로 대체하는 단위 테스트 도구

```java
import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.context.ApplicationContext;

import static org.junit.jupiter.api.Assertions.assertNotNull;

@SpringBootTest
class SpringBootIntellijApplicationTests {

    // 스프링이 만든 ApplicationContext를 주입받는다
    @Autowired
    private ApplicationContext context;

    @Test
    void contextLoads() {
        assertNotNull(context.getBean(GasolineCar.class));
        assertNotNull(context.getBean(Driver.class));
    }
}

```
