## 1.1 IntelliJ 기반 프로젝트 구성

### 1.1.1 새 프로젝트 시작
- IntelliJ IDEA > File > New > Project...

### 1.1.2 Spring Boot 선택
- 생성창 좌측의 Generators 목록 중에서 Spring Boot 선택

### 1.1.3 프로젝트 기본 정보 입력
- Name: `spring-board`
- Location(프로젝트를 생성할 폴더 선택): `c/bebc25/springboot-yong`(예시)
- Language: Java
- Type: Gradle - Groovy
- Group: `net.likelion.bebc25`
- Artifact: `spring-board`
- Package name: `net.likelion.bebc25.board`
- JDK(컴파일러/런타임): 25 Eclipse Temurin
- Java(코드내의 Java 문법 버전): 25
- Packaging: Jar
- Configuration: Properties
- Next 클릭

### 1.1.4 스프링 부트 버전 및 의존성 라이브러리 선택
- Spring Boot: 4.1.0
- Dependencies
    - Developer Tools > `Spring Boot DevTools`: 코드 변경 시 서버 자동 재시작 등의 개발 편의 기능 제공
    - Developer Tools > `Lombok`: 반복 적으로 작성하는 코드를 자동으로 생성하는 기능 제공
    - Web > `Spring Web`: 웹 애플리케이션 개발 지원 (Tomcat 서버 내장, MVC 기반의 웹개발 지원)
    - Template Engines > `Thymeleaf`: 서버 사이드 HTML 뷰 템플릿 동적 렌더링 지원
    - I/O > `Validation`: 데이터 입력 유효성 검증 지원

- Create 버튼 클릭

### 1.1.9 실시간 변경 감지 및 자동 재시작 설정
- Spring Boot DevTools 주요 기능
    - 스프링 부트 애플리케이션 개발 단계에서 자바 코드나 정적 리소스를 수정할 때, 매번 서버를 재시작하고 웹 브라우저를 새로고침해야 하는 번거로움을 줄여 개발 생산성을 높이는 도구
    - 자동 재시작 (Automatic Restart): 자바 소스코드(.java) 수정 시 클래스로더 교체를 통해 내장 톰캣 서버를 재시작하는 기능
    - 라이브 리로드 (LiveReload): 정적 자원이 갱신되었을 때 웹 브라우저에 실시간 리프레시 신호를 전송하여 화면을 자동으로 새로고침해 주는 기능
    - 템플릿/정적 자원 캐싱 비활성화: 타임리프 템플릿 파일이나 CSS, JS 파일이 수정되었을 때 화면에 즉시 로딩되도록 뷰 템플릿 엔진의 내부 리소스 캐시 설정을 기본적으로 무효화하는 기능

#### 1) 자동 재시작 설정
- IntelliJ IDEA는 소스 코드 수정 시 자동으로 컴파일을 빌드하는 기능이 꺼져 있으므로, 코드를 저장하더라도 DevTools가 변경을 감지하지 못함
- 따라서 코드가 갱신되는 즉시 백그라운드에서 프로젝트 컴파일을 강제 실행하고, 서버가 구동 중인 상태에서도 이를 즉각 반영할 수 있도록 아래 두 가지 설정을 적용
- IntelliJ IDEA > File > Settings...
    - Build, Execution, Deployment > Compiler 선택 후 우측 `Build project automatically` 옵션 체크 후 `Apply` 버튼 클릭
    - Advanced Settings 선택 후 스크롤을 내리면 Compiler 하위의 `Allow auto-make to start even if developed application is currently running` 옵션 체크 후 `OK` 클릭

#### 2) 라이브 리로드 설정
- 서버 설정
    - `src/main/resources/application.properties` 파일을 열고 아래 설정을 추가
      ```properties
      spring.devtools.livereload.enabled=true
      ```
        - 주의사항: Spring Boot 4.x 스펙부터는 보안 상의 이유로 라이브 리로드 서버의 활성화 기본값(Default)이 false로 변경되었으며, 해당 옵션이 Deprecated(향후 삭제 예정) 처리됨

- 웹 브라우저 설정
    - 원리: 스프링 부트 백엔드 내부에서 기동된 LiveReload 웹소켓 서버(포트 35729)와 크롬 브라우저의 확장 프로그램이 양방향 소켓 통신망을 형성함. 이후 컴파일이 감지되어 리소스가 변경되면, 서버가 소켓망을 통해 브라우저 측으로 새로고침 명령을 송신하고 확장 프로그램이 이를 수신하여 브라우저 화면을 강제로 갱신함
    - 설치: 크롬 웹스토어에서 [LiveReload Reborn](https://chromewebstore.google.com/detail/livereload-reborn/kdlcalkcmchabpgfhmjannmkbppggkck) 확장 프로그램을 추가
    - 활성화: 스프링 부트 서버 구동 후 웹 브라우저로 `http://localhost:8080`에 접속한 상태에서, 브라우저 우측 상단 툴바의 LiveReload Reborn 아이콘을 클릭하여 활성화 (아이콘의 우측 상단에 초록색의 동그라미가 생김)
    - 적용 확인: HTML 또는 CSS 파일을 수정하고 컴파일이 감지되면 서버를 직접 껐다 켜지 않아도 웹 브라우저가 알아서 실시간 새로고침되어 반영되는지 확인

# 2. HTTP 프로토콜
HTTP(HyperText Transfer Protocol)는 웹 브라우저와 웹 서버 간 텍스트 기반으로 데이터를 송수신하기 위한 프로토콜

## 2.1 주요 특징
- TCP 기반 프로토콜
    - 클라이언트와 서버가 연결을 수립한 후 메시지를 교환
    - 데이터를 패킷 단위로 전송하며, 수신 측에서 데이터를 검증한 뒤 응답을 보냄
    - 데이터 누락 시 재전송 요청을 처리하여 신뢰성이 높음
    - 주로 HTTP, FTP, SMTP 등 신뢰성이 중요한 통신에서 사용됨

### 2.1.1 장점
- 신뢰성: TCP를 기반으로 하여 데이터 전달의 신뢰성이 높음
- 범용성: 전 세계적으로 가장 널리 사용되는 웹 통신 프로토콜

### 2.1.2 참고
- HTTP/2와 HTTP/3는 기존 HTTP/1.x의 비효율성을 개선하기 위해 다중화, 헤더 압축, UDP 기반 전송 등의 기술을 도입

## 2.2 동작 방식
### 2.2.1 클라이언트의 요청 (Request)
- 클라이언트(주로 웹 브라우저)가 HTTP 요청 메시지를 서버로 전송
- 요청에는 필요한 자원(URL)이나 작업(GET, POST 등 요청 메서드 포함)이 포함됨

### 2.2.2 서버의 응답 (Response)
- 서버는 요청을 분석하여 필요한 작업을 수행
    + 파일 읽기, 데이터베이스 조회, 외부 시스템 연동 등
- 작업 결과를 바탕으로 응답 메시지를 생성하여 클라이언트로 전송

### 2.2.3 웹 브라우저의 처리
- 클라이언트는 서버 응답 데이터를 받아 파싱한 후 화면에 출력
- 서버의 응답을 받은 후 클라이언트와 서버 간 연결이 해제됨 (HTTP/1.x의 경우 기본적으로 비연결형(Connectionless) 프로토콜)

### 2.3.1 요청 라인 (Request Line)
```
GET /posts HTTP/1.1
POST /posts HTTP/1.1
PATCH /posts/5 HTTP/1.1
```

- 요청 대상(Request Target): URL 경로
- HTTP 버전(HTTP Version): HTTP/1.1 또는 HTTP/2 등

```
If-None-Match: W/"191-19a1fcb18fe"
Referer: localhost:8080/index.html
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/141.0.0.0 Safari/537.36
```

- `Accept: text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,image/apng,*/*;q=0.8`
    + 클라이언트가 선호하는 응답 콘텐츠 타입과 품질 지수(q값: 0~1, 생략시 1, 1이 가장 높은 선호도)
- `Accept-Encoding: gzip, deflate, br, zstd`
    + 클라이언트가 지원하는 압축 방식
- `Accept-Language: ko,en-US;q=0.9,en;q=0.8`
    + 클라이언트가 선호하는 언어와 우선순위
- `Cache-Control: no-cache`
    + 캐시 동작 제어
    + no-cache: 캐시된 응답을 사용하기 전 서버에 검증 요청 (항상 재검증, 304 Not Modified이면 캐시 사용, 변경되었으면 새 데이터 다운로드)
    + no-store: 응답을 캐시에 저장하지 않음
    + max-age=3600: 캐시 유효 시간(초 단위)
- `Connection: keep-alive`
    + 연결 유지 여부 (keep-alive: 연결 유지, close: 연결 종료)
- `Host: localhost:8080`
    + 요청을 보내는 서버의 도메인 주소와 포트
- `If-Modified-Since: Mon, 13 Jul 2026 09:13:15 GMT`
    + 캐시된 리소스의 Last-Modified 시간을 전송, 서버가 변경되지 않았으면 304 Not Modified 응답
- `If-None-Match: W/"191-19a1fcb18fe"`
    + 캐시된 리소스의 ETag 값을 전송, 서버가 변경되지 않았으면 304 Not Modified 응답
- `Referer: http://127.0.0.1:8080/index.html`
    + 요청이 시작된 이전 페이지의 URL (철자 주의: Referer는 실제로는 Referrer지만 명세상 오타로 Referer로 표기)
- `User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/141.0.0.0 Safari/537.36`
    + 클라이언트의 브라우저 및 OS 정보

### 2.3.3 요청 바디 (Request Body)
```
title=hello&content=hi&author=namu
```

- GET 요청: 바디 없음 (데이터는 쿼리 파라미터로 URL에 포함하여 전송)
- POST, PUT, PATCH, DELETE 요청: JSON, XML, form-data 등 다양한 형식의 데이터 전송 가능

### 2.4.1 상태 라인 (Status Line)
```
HTTP/1.1 200 OK
HTTP/1.1 201 Created
HTTP/1.1 404 Not Found
HTTP/1.1 500 Internal Server Error
```
- HTTP 버전(HTTP Version): HTTP/1.1 또는 HTTP/2 등
- 상태 코드(Status Code): 200(성공), 404(찾을 수 없음), 500(서버 에러) 등
- 이유 구문(Reason Phrase): OK, Created, Not Found 등 응답 상태를 간단한 텍스트로 표현

#### 주요 응답 상태 코드

##### 2xx (성공)
- `200 OK`: 요청이 성공적으로 처리됨, 응답 본문에 요청된 리소스가 포함됨
- `201 Created`: 요청이 성공했으며 새로운 리소스가 생성됨 (일반적으로 POST 요청)
- `204 No Content`: 요청은 성공했지만 응답 본문이 없음 (일반적으로 DELETE 요청)

##### 3xx (리디렉션)
- `301 Moved Permanently`: 요청한 리소스가 영구적으로 이동됨, 새 URL은 Location 헤더에 포함
- `302 Found`: 요청한 리소스가 임시로 이동됨
- `304 Not Modified`: 리소스가 변경되지 않아 캐시된 버전을 사용해도 됨

##### 4xx (클라이언트 오류)
- `400 Bad Request`: 잘못된 요청 구문으로 서버가 처리할 수 없음
- `401 Unauthorized`: 인증이 필요함, 로그인 정보가 없거나 만료됨
- `403 Forbidden`: 인증은 됐지만 권한이 없어 거부됨
- `404 Not Found`: 요청한 리소스를 찾을 수 없음
- `405 Method Not Allowed`: 요청한 HTTP 메서드가 허용되지 않음

##### 5xx (서버 오류)
- `500 Internal Server Error`: 서버 내부 오류, 예상치 못한 문제 발생

### 2.4.2 응답 헤더 (Response Headers)
```
Cache-Control: no-store
Last-Modified: Mon, 13 Jul 2026 01:43:37 GMT
Accept-Ranges: bytes
Content-Type: text/html
Date: Mon, 13 Jul 2026 01:56:52 GMT
Keep-Alive: timeout=60
Connection: keep-alive
Transfer-Encoding: chunked
```

- `Cache-Control: no-store`
    + 캐시 동작 제어 (no-store: 응답 결과를 브라우저 및 중간 캐시 서버에 임시 저장하지 않도록 제한)
- `Last-Modified: Mon, 13 Jul 2026 01:43:37 GMT`
    + 리소스가 마지막으로 수정된 시간 (If-Modified-Since와 함께 캐시 검증에 사용)
- `Accept-Ranges: bytes`
    + 서버가 `부분 요청(Partial Request)`을 지원함
    + 클라이언트가 `Range: bytes=1000-1999` 형식으로 바이트 단위의 범위 요청 가능
- `Content-Type: text/html`
    + 응답 바디의 미디어 타입
- `Date: Mon, 13 Jul 2026 01:56:52 GMT`
    + 응답이 생성된 시간
- `Keep-Alive: timeout=60`
    + keep-alive 연결의 최대 유지 시간(60초)
- `Connection: keep-alive`
    + 연결 유지 여부 (keep-alive: 연결 유지, HTTP/1.1에서는 기본값)
- `Transfer-Encoding: chunked`
    + 데이터 전송 인코딩 방식 (chunked: 청크 단위로 분할 전송)

### 2.4.3 응답 바디 (Response Body)
```html
<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8">
  <title>스프링 게시판 - 목록</title>
  ...
</head>
<body>
  ...
</body>
</html>
```

- 성공 응답: 요청한 데이터를 응답 바디에 JSON, XML, HTML 등의 형식으로 반환
- 에러 응답: 에러 메시지와 함께 상태 정보 반환

## 2.5 HTTP의 특징

### 2.5.1 비연결성(Connectionless)
- 클라이언트가 요청을 보내고 서버가 응답하면 상호 연결을 해제함
- 서버는 클라이언트가 누구인지 신경 쓰지 않고 요청 정보만 분석해 적절한 응답을 만드는 데만 집중함
- 서버 구현이 단순해져 웹 발전에 큰 역할을 함

### 2.5.2 무상태(Stateless)
- 서버는 클라이언트의 요청 정보만 가지고 응답을 생성하며, 클라이언트의 상태를 기억하지 않음
- 요청한 사용자가 누구인지, 이전에 어떤 작업을 요청했는지를 관리하지 않음
    + 웹이 발전하면서 클라이언트 정보를 저장할 필요가 생겼고, 이를 위해 Cookie, Session, JWT 등의 저장 방식이 도입됨

## 2.6 HTTP 주요 메서드
- HTTP 메서드는 클라이언트가 서버에 요청을 보낼 때, 대상 자원(Resource)에 대해 어떠한 행위(조회, 등록, 수정, 삭제)를 수행하고자 하는지 알려주는 기능
- 메서드의 성격을 규정하는 주요 특징 속성
    - 안전성 (Safe Methods): 요청을 여러 번 호출해도 대상 자원의 상태를 변경하지 않는 안전한 성질 (예: GET)
    - 멱등성 (Idempotent Methods): 동일한 요청을 한 번 실행한 것과 여러 번 연속해서 실행한 결과가 서버 리소스 상태 기준으로 항상 동일한 성질 (예: GET, PUT, DELETE)
    - 요청 바디 존재 여부: 전송할 데이터(Payload)를 URL에 얹느냐(GET) 혹은 HTTP 메시지 바디에 실어 전송하느냐(POST, PUT, PATCH)의 차이

### 2.6.1 GET
- 서버로부터 자원을 가져올 때 사용
    + 게시글 목록 조회
    + 게시글 상세 조회
    + 회원 정보 조회

### 2.6.2 POST
- 서버로 데이터를 보낼 때 사용
    + 게시글 등록
    + 회원 등록

### 2.6.3 PUT
- 서버 데이터 한 건의 전체 항목 수정 시 사용
    + 특정 게시글의 모든 항목 수정
    + 특정 회원의 회원 정보 전체 수정

### 2.6.4 PATCH
- 서버 데이터 한 건의 일부 항목 수정 시 사용
    + 특정 게시글의 제목 수정
    + 특정 회원의 비밀번호 수정

### 2.6.5 DELETE
- 서버의 데이터 한 건을 삭제할 때 사용
    + 게시글 삭제
    + 회원 삭제

# 3. 3티어 아키텍처와 MVC 패턴 설계

## 3.1 3티어 아키텍처의 이해

### 3.1.1 3티어 아키텍처의 개념
- 3티어 아키텍처는 전체 웹 시스템을 물리적 및 독립적인 세 개의 하드웨어 서버(혹은 네트워크 영역)로 분할하여 구축하고 운영하는 인프라 아키텍처 설계 패턴
- 각 티어는 독립적인 시스템 프로세스로 구동되며, 통신망을 통해 데이터를 주고받음으로써 특정 영역의 장애가 전체로 확산되는 것을 방지하고 독립적인 가동 환경을 형성

### 3.1.2 3티어 아키텍처의 물리적 구성 요소
#### 1) 프레젠테이션 계층 (Presentation Tier)
- 사용자 기기(컴퓨터, 스마트폰 등) 화면에 보여지는 영역
- 웹 브라우저(크롬, 사파리 등)가 이에 해당하며, 서버로부터 전달받은 파일(HTML, CSS, JavaScript)을 파싱해서 화면에 그리고 사용자의 입력을 서버로 전달하는 역할
- 사용자 기기의 성능을 직접 활용하며 서버와는 네트워크를 통해 연결

#### 2) 애플리케이션 계층 (Application Tier / Business Tier)
- 사용자가 요청한 회원가입이나 글쓰기 등 프로그램의 실제 핵심 기능을 실행하고 처리하는 영역
- 자바 코드로 작성한 프로그램이 가동되는 서버 환경으로, 스프링 부트 애플리케이션이 실행되는 곳이 이에 해당
- 입력된 데이터를 검증하고 상황에 맞게 계산 및 판단을 내리며, 데이터베이스에 필요한 데이터를 요청하거나 저장하도록 조율하는 역할

#### 3) 데이터 계층 (Data Tier)
- 프로그램이 다루는 모든 회원 정보나 게시글 데이터 등을 안전하게 저장하고 관리하는 데이터 창고 영역
- 데이터베이스 관리 소프트웨어(MySQL, Oracle 등)가 실행되는 전용 컴퓨터 환경이 이에 해당
- 애플리케이션 계층의 심부름(데이터 조회, 저장 등) 요청을 받아 실제 테이블에서 데이터를 꺼내거나 새로 집어넣는 역할

### 3.1.3 물리적 분리의 이점
- 자원 활용의 유연성: 특정 계층의 부하가 가중될 경우 해당 티어의 서버 장비만 추가로 증설하여 확장(Scale-out)하기 수월
- 유지보수 제어: 소프트웨어 구동 환경이 물리적으로 격리되어 있어 특정 서버의 운영체제나 소프트웨어 버전을 업그레이드해도 타 서버에 미치는 영향이 없음
- 인프라 보안성 제고: 핵심 자산인 데이터베이스 서버를 외부 망에 노출하지 않고 사설 네트워크에 은닉하며, 오직 애플리케이션 서버(WAS)를 통해서만 접근을 허용하여 인프라 취약성을 제어

## 3.2 Model 1과 Model 2 아키텍처
웹 애플리케이션 아키텍처는 제어 흐름과 뷰의 분리 수준에 따라 Model 1과 Model 2 구조로 구분

### 3.2.1 Model 1 아키텍처
- 특징: 하나의 웹 문서 파일(HTML 등) 내부에 화면을 그리는 디자인 코드와 데이터를 다루는 자바 프로그램 코드가 구분 없이 한곳에 섞여서 구동되는 고전적인 구조
- 장점: 구조가 복잡하지 않아 소규모 개발 시 빠른 동작 검증이 가능
- 단점: 단일 파일 안에 프로그램 연산 흐름과 화면 요소가 뒤섞여 있어 페이지가 커질수록 유지보수가 어려우며, 디자이너와 개발자 간의 역할 분담이 어려워 협업 생산성을 저하함

### 3.2.2 Model 2 아키텍처 (MVC 패턴)
- 특징: 클라이언트의 요청 처리(Controller), 비즈니스 로직(Model), 화면 출력(View) 역할을 명확히 격리한 구조
- 컨트롤러(Controller) 역할을 담당하는 서버의 자바 프로그램이 요청을 수신하여 연산을 실행하고, 그 결과를 Thymeleaf 같은 뷰 템플릿에 주입하여 화면을 동적으로 완성
- 장점: 프레젠테이션 영역과 비즈니스 영역의 결합도가 낮아 역할 분담 및 유지보수성 향상 가능. 스프링 MVC 프레임워크 동작 구조의 모태가 됨

## 3.3 MVC 패턴
- 사용자 인터페이스, 데이터 및 제어 로직을 상호 분리하여 개발하는 소프트웨어 디자인 패턴
- Model: 애플리케이션의 데이터 및 핵심 비즈니스 로직을 담당하는 영역
- View: Model의 데이터를 바탕으로 프리젠테이션 티어에 제공할 View(HTML, JSON 등)를 만드는 영역
- Controller: 사용자의 입력 요청을 수신하여 분석하고, 필요한 비즈니스 로직을 호출하여 Model의 상태를 변경한 뒤 결과를 View에 전달하는 조율 영역
- 식당의 운영 구조와 비교
    - Model은 주방의 요리사, 실질적인 음식 제조(비즈니스 로직 수행 및 데이터 처리)를 담당
    - View는 메뉴판 및 식탁의 음식 플레이팅, 손님이 직접 보고 소비하는 최종 시각 화면을 의미
    - Controller는 홀의 서빙 직원, 손님의 주문(요청 수신)을 주방(Model)에 전달하고, 조리된 음식을 손님 식탁(View)으로 서빙하는 중개 및 조율을 담당

## 3.4 계층별 클래스 설계 기법 (Layered Architecture)
웹 애플리케이션의 유지보수성 향상을 위해 각 클래스를 역할에 따라 자바 소스코드 수준에서 다음과 같이 세 개의 계층(Layer)으로 나누어 설계

### 3.4.1 컨트롤러 계층 (Controller Layer)
- 클라이언트의 웹 요청을 가장 먼저 수신하여 데이터 검증 등을 처리하고, 서비스 계층을 호출한 뒤 결과 화면을 반환하는 클래스 영역

### 3.4.2 서비스 계층 (Service Layer)
- 프로그램의 핵심 비즈니스 규칙을 구현하고 연산하는 영역이며, 여러 데이터베이스 연동 작업들의 안전한 실행 단위(트랜잭션)를 관리하는 클래스 영역

### 3.4.3 레포지토리 계층 (Repository Layer)
- 데이터베이스와 직접 연결되어 테이블에 데이터를 저장하고 가져오는 등 실제 물리적 데이터 접근을 전담하는 클래스 영역

### 3.4.4 클라이언트의 요청이 서버로 전달되는 과정
- 클라이언트 -> 애플리케이션 서버 (컨트롤러 -> 서비스 -> 레포지토리) -> 데이터베이스 서버

### 3.4.5 서버의 응답이 클라이언트로 전달되는 과정
- 데이터베이스 서버 -> 애플리케이션 서버 (레포지토리 -> 서비스 -> 컨트롤러 -> 뷰) -> 클라이언트

## 3.5 계층 간 데이터 전달 객체의 분류
컨트롤러, 서비스, 레포지토리 계층 간에 정보를 교환하거나 데이터베이스의 결과를 담아둘 때 활용하는 데이터 전달 목적용 전용 클래스

### 3.5.1 DTO (Data Transfer Object)
- 목적: 소프트웨어 계층 간에(예: Controller와 Service) 데이터를 전송하기 위한 데이터 전달용 객체
- 특징: 비즈니스 로직을 갖지 않고 오직 읽기/쓰기 필드(getter/setter 혹은 자바 record)만을 보유하는 단순 데이터 홀더 객체

### 3.5.2 Entity
- 목적: 데이터베이스 테이블과 1대1로 직접 대응되어 매핑되는 영속성 객체
- 특징
    - 데이터베이스 내에서 고유하게 식별할 수 있는 키(ID)를 가지며, 테이블 레코드 하나의 데이터를 담고 있음
    - 주로 서비스 계층과 레포지토리 계층 내부에서 비즈니스 로직을 수행하고 데이터를 처리할 때 활용함
    - 보안 및 설계 격리를 위해 외부 컨트롤러 계층까지 직접 노출하지 않으며, 서비스에서 컨트롤러에 데이터를 반환할 때는 전용 DTO 객체로 변환해 전달하는 것이 표준 설계

### 3.5.3 DAO (Data Access Object)
- 목적: 데이터베이스와 직접 연결되어 데이터를 저장하고 조회하는 SQL 쿼리를 실행해 주는 데이터 접근 전용 객체

## 3.6 프로젝트 패키지 구조 설계
- 회원 기능(Member)과 게시글 기능(Post)처럼 두 개 이상의 업무 도메인이 공존할 때 설계하는 프로젝트 패키지 구조는 크게 두 가지 방식으로 분류함

### 3.6.1 계층 기반 패키지 구조 (Package by Layer)
- 기술적 역할(Controller, Service, Repository 등)을 최상위 패키지로 삼고, 그 하위에 모든 도메인 클래스들을 모으는 전통적인 방식
```
net.likelion.bebc25.board
├── controller
│   ├── PostController.java
│   └── MemberController.java
├── service
│   ├── PostService.java
│   └── MemberService.java
├── repository
│   ├── PostRepository.java
│   └── MemberRepository.java
├── entity
│   ├── Post.java
│   └── Member.java
└── dto
    ├── PostDto.java
    └── MemberJoinDto.java
```
- 특징: 프로젝트 규모가 작을 때 계층별 분류가 명확하여 구조 파악이 용이하나, 도메인이 늘어나면 패키지 내부에 많은 클래스가 만들어져서 작업 시 패키지를 크게 오가며 찾아야 하는 불편함이 발생

### 3.6.2 도메인 기반 패키지 구조 (Package by Domain)
- 업무 기능(post, member)을 최상위 패키지로 세분화하고, 각 도메인 패키지 내부에서 하위 기술 계층 패키지를 각각 격리하여 소유하는 방식
```
net.likelion.bebc25.board
├── post
│   ├── controller
│   │   └── PostController.java
│   ├── service
│   │   └── PostService.java
│   ├── repository
│   │   └── PostRepository.java
│   ├── entity
│   │   └── Post.java
│   └── dto
│       └── PostDto.java
└── member
    ├── controller
    │   └── MemberController.java
    ├── service
    │   └── MemberService.java
    ├── repository
    │   └── MemberRepository.java
    ├── entity
    │   └── Member.java
    └── dto
        └── MemberJoinDto.java
```
- 특징: 각 업무 도메인 단위의 강한 응집도를 보장하며, 향후 특정 기능을 마이크로서비스(MSA) 형태로 추출하고 격리하기 유리함

# 4. 컨트롤러 만들기

## 4.1 Controller 어노테이션 정의 및 응답 데이터 생성
- HTTP 요청을 수신하여 비즈니스 로직을 연결하고, 그 결과를 클라이언트에게 직접 문자열이나 HTML 데이터로 반환하는 기초 웹 컨트롤러 설계 기법

### 4.1.1 컨트롤러 정의 및 응답용 어노테이션
- `@Controller`
    - 클래스 상단에 명시하여 해당 클래스를 스프링 컨테이너가 관리하는 웹 요청 핸들러(컨트롤러) 빈으로 등록함
- `@ResponseBody`
    - 메서드 상단 혹은 반환 타입 앞에 부여하여, 메서드가 리턴하는 값(HTML 텍스트, DTO 등)을 HTTP 응답 메시지의 바디(Body) 영역에 직접 기입하여 전송함

### 4.1.2 웹 요청 경로 매핑용 어노테이션 및 주요 속성
- `@RequestMapping`
    - 클래스나 메서드에 부여하여 해당 컨트롤러가 처리할 공통 URL 경로 범위를 지정
    - 주요 속성
        - `value` 또는 `path`: 매핑할 URL 주소 경로를 지정
        - `method`: 처리할 HTTP 메서드 방식을 직접 지정 (생략할 경우 모든 메서드를 다 수용)

      ```java
      @RequestMapping(value = "/board/list", method = RequestMethod.GET)
      @ResponseBody
      public String getBoardList() {
          return "<html>...</html>";
      }
      ```

- `@GetMapping` 및 `@PostMapping`
    - 각각 HTTP GET, POST 요청을 수신할 세부 URL 매핑 경로를 단축형으로 지정할 때
    - 주요 속성
        - `value` 또는 `path`: 세부 URL 주소 경로를 지정

      ```java
      @GetMapping(value = "/board/list")
      public String getBoardList() {
          return "<html>...</html>";
      }
      ```

      ```java
      @PostMapping(value = "/board/write")
      public String writePost() {
          return "redirect:/board/list";
      }
      ```

#### 1) 요청 매핑 및 데이터 직접 응답 예시
- 클래스에 공통 경로 지정을 부여하고, 메서드 레벨에서 HTML 문자열 데이터를 직접 반환하도록 ResponseBody를 결합 구현함

```java
package net.likelion.bebc25.board.controller;

import org.springframework.stereotype.Controller;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.ResponseBody;

@Controller
@RequestMapping("/board01")
public class BoardSimpleController {

    @GetMapping(value = "/index.html")
    @ResponseBody
    public String welcome() {
        return "<h1>안녕하세요</h1><p>스프링부트 실습 환경에 오신 것을 환영합니다.</p>";
    }
}
```

- 뷰 반환 제어 방식(Forward)과 리다이렉트(Redirect)의 데이터 전송 흐름 비교:
    - 뷰 이름 반환 (Forward 방식): 서버 내부에서 해당 뷰 객체로 흐름을 양도하여 직접 HTML을 구성한 뒤 응답함. 브라우저의 URL 주소창은 변경되지 않고 요청 경로를 그대로 유지함
    - 리다이렉트 (Redirect 방식): 응답 헤더에 302 상태 코드와 Location 경로를 담아 전송하여 브라우저가 새 주소로 다시 GET 요청을 보내도록 유도함. 브라우저의 URL 주소창이 최종 목적지 경로로 변경되므로 새로고침 시 중복 처리(게시글 이중 등록 등) 문제를 제어하는 데 수월함

## 4.2 HTTP 파라미터 매핑 방법
- 클라이언트가 전달한 웹 요청 데이터를 컨트롤러 메서드의 매개변수로 수집
    - @RequestParam: HTTP 요청 파라미터(예: ?name=value)를 1대1 매핑하여 수집
    - @PathVariable: URI 경로 자체에 포함된 변수 값(예: /posts/{id}에서 id)을 추출하여 바인딩
    - @ModelAttribute: HTML 폼 데이터를 DTO 객체 필드에 일괄 바인딩

```java
package net.likelion.bebc25.board.controller;

import org.springframework.stereotype.Controller;
import org.springframework.ui.Model;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PathVariable;
import java.util.ArrayList;
import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;
import net.likelion.bebc25.board.dto.BoardPostDto;

@Controller
public class BoardController {

    private final Map<Long, BoardPostDto> database = new ConcurrentHashMap<>();

    // 루트 경로 요청 시 게시글 목록으로 자동 리다이렉트
    @GetMapping("/")
    public String index() {
        return "redirect:/board/posts";
    }

    // 게시글 목록 조회 및 뷰 반환
    @GetMapping("/board/posts")
    public String listPosts(Model model) {
        model.addAttribute("posts", new ArrayList<>(database.values()));
        return "board/list";
    }

    // 경로 변수(@PathVariable)를 이용한 단건 조회 예시
    @GetMapping("/board/posts/{id}/edit")
    public String editForm(@PathVariable("id") Long id, Model model) {
        BoardPostDto postDto = database.get(id);
        if (postDto == null) {
            throw new IllegalArgumentException("존재하지 않는 게시글 번호입니다: " + id);
        }
        model.addAttribute("post", postDto);
        return "board/form";
    }
}
```

# 5. 타임리프 뷰 렌더링

## 5.1 타임리프 기본 문법
- HTML 코드의 본래 마크업 규격을 유지하면서 서버의 스프링 데이터를 주입하여 화면을 렌더링하는 뷰 템플릿 엔진
    - th:text: 서버가 전달한 문자열 데이터를 태그 텍스트로 렌더링
    - th:each: 전달받은 컬렉션 리스트의 크기만큼 반복하여 HTML 요소를 출력
    - th:if / th:unless: 조건 조건식의 참/거짓 여부에 따라 HTML 요소를 선택적으로 노출
    - th:object와 th:field: 컨트롤러에서 바인딩한 DTO 객체 모델을 수신하여 form 입력창의 name 및 value 속성을 자동 연결

## 5.2 공통 레이아웃 설계 및 컴포넌트 재사용
- 웹 페이지의 헤더, 네비게이션, 푸터 등 반복되는 공통 마크업을 별도 레이아웃 파일로 모듈화하여 재사용성 향상
- th:fragment 속성으로 공통 템플릿 조각을 정의하고, 적용할 화면에서 th:replace 또는 th:insert 속성을 가동해 끼워 넣는 설계 적용

# 6. 데이터 검증 및 예외 처리

## 6.1 Controller 계층에서의 입력값 검증 필요성 및 @Valid 활용
- 클라이언트로부터 전달받은 불완전하거나 유해한 데이터가 내부 비즈니스 로직이나 저장소로 전파되는 것을 방지하기 위해 프레젠테이션 계층에서 1차 검증 강제 적용 필요
- DTO 필드에 검증 어노테이션(@NotBlank, @Size 등)을 선언하고, 컨트롤러 메서드 매개변수 앞에 @Valid를 부여하여 유효성 감지 자동 위임

```java
package net.likelion.bebc25.board.dto;

import jakarta.validation.constraints.NotBlank;
import jakarta.validation.constraints.Size;

public class BoardPostDto {
    private Long id;

    @NotBlank(message = "제목은 필수 입력 항목입니다.")
    @Size(max = 100, message = "제목은 100자 이하로 입력해야 합니다.")
    private String title;

    @NotBlank(message = "내용은 필수 입력 항목입니다.")
    private String content;

    @NotBlank(message = "작성자는 필수 입력 항목입니다.")
    @Size(min = 2, max = 10, message = "작성자 이름은 2자 이상 10자 이하여야 합니다.")
    private String author;
}
```

## 6.2 BindingResult 객체를 이용한 검증 에러 및 예외 처리
- BindingResult는 데이터 검증 도중 규격 위반 오류가 발견되었을 때, 프로그램이 예외를 발생시키며 중단되는 대신 검증 실패 상세 정보를 수집하여 전달하는 전용 인터페이스
- 컨트롤러는 BindingResult.hasErrors() 메서드를 통해 오류 유무를 확인하고, 에러가 존재할 시 비즈니스 처리를 진행하지 않고 본래의 입력 폼 화면으로 회귀 분기 처리
- 공항 입국 심사대에서 기재 오류가 발견되었을 때 바로 추방하지 않고 수정이 필요한 부분을 짚어 서류 보완 요청을 내려주는 처리에 비유 가능
- 전역 예외 처리를 담당하는 @ControllerAdvice 어노테이션 기반 클래스를 설계하여 다수의 컨트롤러에서 예기치 않게 발생하는 에러를 일괄 통합 제어

- BindingResult와 타임리프 에러 매핑 동작 메커니즘:
    - 스프링 MVC는 DTO 검증 실패 시 생성된 검증 에러 정보를 BindingResult 객체에 저장하여 뷰 템플릿에 모델 속성으로 자동 전달
    - 타임리프 뷰 내에서는 th:object="${post}"로 선언된 객체의 에러 여부를 #fields.hasErrors('title') 식으로 검출 가능하며, th:errors="*{title}"을 통해 BindingResult에 저장된 해당 필드의 에러 메시지를 태그 텍스트로 동적 출력 처리

```html
<form th:action="${post.id == null} ? @{/board/posts/new} : @{/board/posts/{id}/edit(id=${post.id})}" th:object="${post}" method="post">
  <div class="form-group">
    <label>제목</label>
    <input type="text" th:field="*{title}">
    <div class="field-error" th:if="${#fields.hasErrors('title')}" th:errors="*{title}">에러 메시지</div>
  </div>
  <div class="form-group">
    <label>작성자</label>
    <input type="text" th:field="*{author}">
    <div class="field-error" th:if="${#fields.hasErrors('author')}" th:errors="*{author}">에러 메시지</div>
  </div>
  <button type="submit">등록</button>
</form>
```

# 7. Spring MVC 웹 테스트와 MockMvc

## 7.1 웹 통합 테스트와 MockMvc 개요
- 웹 통합 테스트: WAS 서버를 물리적으로 구동하지 않고도 컨트롤러가 HTTP 요청 주소를 받아 처리하고 뷰 템플릿이 올바르게 반환 및 렌더링 되는지 모의 환경을 빌드하여 일괄 검증하는 기술
- MockMvc: 서블릿 컨테이너를 가동하는 과정 없이 모의 서블릿 환경을 구성하여 컨트롤러의 웹 요청과 응답을 빠르게 시뮬레이션 및 검증하는 테스팅 전용 모킹 엔진

## 7.2 MockMvc 웹 요청 검증 실습
- @SpringBootTest와 @AutoConfigureMockMvc 어노테이션을 결합하여 가상으로 API 및 뷰 컨트롤러 요청을 검증하는 자바 소스 코드를 구성

```java
package net.likelion.bebc25.board.controller;

import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.get;
import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.post;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.model;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.redirectedUrl;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.status;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.view;

import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.webmvc.test.autoconfigure.AutoConfigureMockMvc;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.test.web.servlet.MockMvc;

@SpringBootTest
@AutoConfigureMockMvc
public class BoardControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    public void listPostsViewTest() throws Exception {
        mockMvc.perform(get("/board/posts"))
                .andExpect(status().isOk())
                .andExpect(view().name("board/list"))
                .andExpect(model().attributeExists("posts"));
    }

    @Test
    public void createPostValidationFailureTest() throws Exception {
        mockMvc.perform(post("/board/posts/new")
                        .param("title", "")
                        .param("author", "")
                        .param("content", "유효하지 않은 빈 요청입니다."))
                .andExpect(status().isOk())
                .andExpect(view().name("board/form"))
                .andExpect(model().hasErrors());
    }

    @Test
    public void indexRedirectTest() throws Exception {
        mockMvc.perform(get("/"))
                .andExpect(status().is3xxRedirection())
                .andExpect(redirectedUrl("/board/posts"));
    }
}
```

# 8. 스프링 부트 웹 개발 생산성 및 모니터링 도구

## 8.1 Spring Boot Actuator를 활용한 운영 환경 모니터링
- Spring Boot Actuator는 웹 서버가 가동된 후, 현재 구동 중인 애플리케이션의 내부 상태(JVM 메모리 점유율, 쓰레드 개수, 데이터베이스 연결 상태 등)를 HTTP 엔드포인트(/actuator)를 통해 정밀하게 진단하고 노출해 주는 운영 지원 도구
- 모니터링 엔드포인트 종류:
    - `/actuator/health`: 애플리케이션 및 연결된 DB의 작동 여부(Liveness) 모니터링
    - `/actuator/metrics`: CPU 점유율, 메모리 사용량, HTTP 요청 누적 통계 모니터링

- Actuator 의존성 추가 설정:

```groovy
dependencies {
    runtimeOnly 'org.springframework.boot:spring-boot-starter-actuator'
}
```

- 웹 모니터링 엔드포인트 전역 개방 설정:
  배포 환경에서는 보안을 위해 노출을 통제하지만, 개발 및 관리망 모니터링을 위해 노출 범위를 설정으로 제어

```properties
# 액추에이터 모니터링 엔드포인트 웹 노출 범위 설정
management.endpoints.web.exposure.include=health,metrics
```
